# 4. 授权类型

要请求访问令牌，客户端从资源所有者获取授权。本规范定义了以下授权许可类型：

* 授权码

* 客户端凭证，以及

* 刷新令牌

它还提供了用于定义其他许可类型的扩展机制。

## 4.1. 授权码许可

授权码许可类型用于获取访问令牌和刷新令牌。

该许可类型使用额外的授权端点，让授权服务器与资源所有者交互以获得资源访问同意。

由于这是基于重定向的流程，客户端必须能够使用资源所有者的用户代理（通常是 Web 浏览器）启动流程，并能够从授权服务器接收重定向。

```
     +----------+
     | Resource |
     |   Owner  |
     +----------+
           ^
           |
           |
     +-----|----+          Client Identifier      +---------------+
     | .---+---------(1)-- & Redirect URI ------->|               |
     | |   |    |                                 |               |
     | |   '---------(2)-- User authenticates --->|               |
     | | User-  |                                 | Authorization |
     | | Agent  |                                 |     Server    |
     | |        |                                 |               |
     | |    .--------(3)-- Authorization Code ---<|               |
     +-|----|---+                                 +---------------+
       |    |                                         ^      v
       |    |                                         |      |
       ^    v                                         |      |
     +---------+                                      |      |
     |         |>---(4)-- Authorization Code ---------'      |
     |  Client |          & Redirect URI                     |
     |         |                                             |
     |         |<---(5)----- Access Token -------------------'
     +---------+       (w/ Optional Refresh Token)

                Figure 3: Authorization Code Flow
```

图 3 中说明的流程包括以下步骤：

(1) 客户端通过将资源所有者的用户代理引导到授权端点来启动流程。客户端包含其客户端标识符、代码挑战（从生成的代码验证器派生）、可选的请求作用域、可选的本地状态，以及授权服务器在授予访问权限（或拒绝）后将用户代理发送回的 重定向 URI。

(2) 授权服务器（通过用户代理）对资源所有者进行认证，并确定资源所有者是授予还是拒绝客户端的访问请求。

(3) 假设资源所有者授予访问权限，授权服务器使用之前提供的重定向 URI（请求中或客户端注册期间提供的）将用户代理重定向回客户端。重定向 URI 包含授权码和客户端之前提供的任何本地状态。

(4) 客户端通过包含上一步收到的授权码以及其代码验证器，向授权服务器的令牌端点请求访问令牌。在发出请求时，客户端会向授权服务器进行认证（如果可以）。客户端包含用于获取授权码的重定向 URI 以进行验证。

(5) 授权服务器在可能时对客户端进行认证，验证授权码，验证代码验证器，并确保收到的重定向 URI 与步骤 (3) 中将用户代理重定向到客户端所使用的 URI 匹配。如果有效，授权服务器会返回访问令牌和可选的刷新令牌。

### 4.1.1. 授权请求

要开始授权请求，客户端通过将参数添加到授权服务器的授权端点 URI 来构建授权请求 URI。客户端最终会将用户代理重定向到此 URI 以启动请求。

客户端对每个授权请求使用一个称为"代码验证器"的唯一高熵加密随机字符串，以防止授权码注入和 CSRF 攻击。客户端首先生成代码验证器，然后派生要包含在授权请求中的"代码挑战"。客户端在令牌端点交换授权码时使用代码验证器，以证明使用授权码的客户端与请求它的客户端是同一个客户端。

客户端通过按照附录 C.1 中的描述将以下参数添加到授权端点 URI 的查询组件来构建请求 URI：

**"response_type"**：必需的。授权端点支持不同的请求和响应参数集。客户端通过使用某个 response_type 值来确定流程类型。本规范定义了值 code，它必须用于表示客户端想要使用授权码流程。

扩展响应类型可以包含空格分隔的（%x20）值列表，其中值的顺序不重要（例如，响应类型 a b 与 b a 相同）。此类复合响应类型的含义由各自的规范定义。

某些扩展响应类型由 [OpenID.Connect] 定义。

如果授权请求缺少 response_type 参数，或者响应类型无法理解，授权服务器必须返回第 4.1.2.1 节中描述的错误响应。

**"client_id"**：必需的。如第 2.2 节中所述的客户端标识符。

**"code_challenge"**：除非满足第 7.5.1 节的特定要求，否则是必需的。从代码验证器派生的代码挑战。

**"code_challenge_method"**：可选的，如果请求中不存在则默认为 plain。代码验证器转换方法为 S256 或 plain。

**"redirect_uri"**：如果为此客户端只注册了一个重定向 URI，则是可选的。如果为此客户端注册了多个重定向 URI，则是必需的。参见第 2.3.2 节。

**"scope"**：可选的。访问请求的作用域，如第 1.4.1 节所述。

**"state"**：可选的。客户端用于维护请求和回调之间状态的 opaque 值。授权服务器在将用户代理重定向回客户端时包含此值。

code_verifier 是为每个授权请求生成的唯一高熵加密随机字符串，使用非保留字符 [A-Z] / [a-z] / [0-9] / "-" / "." / "_" / "~"，最小长度为 43 个字符，最大长度为 128 个字符。

客户端临时存储 code_verifier，并计算它将在授权请求中使用的 code_challenge。

code_verifier 的 ABNF 如下：

```
code-verifier = 43*128unreserved
unreserved = ALPHA / DIGIT / "-" / "." / "_" / "~"
ALPHA = %x41-5A / %x61-7A
DIGIT = %x30-39
```

如果 code_verifier 在授权请求中不暴露，客户端应使用不暴露 code_verifier 的代码挑战方法。否则，能够读取授权请求的攻击者（参见 [RFC9700] 中的攻击者 A4）可以破坏此机制提供的安全性。目前，S256 是唯一这样的方法。

注意：code verifier 应具有足够的熵以使其值不可猜测。建议使用合适的随机数生成器的输出创建 32 字节序列。然后将字节序列 base64url 编码以产生 43 字节的 URL 安全字符串，用作 code verifier。

然后客户端通过使用以下转换之一对代码验证器进行转换来创建 code_challenge：

**S256**
```
code_challenge = BASE64URL-ENCODE(SHA256(ASCII(code_verifier)))
```

**plain**
```
code_challenge = code_verifier
```

如果客户端能够使用 S256，则必须使用 S256，因为 S256 是服务器上必须实现的（MTI）。如果客户端由于某些技术原因（例如，没有可用哈希函数的受限环境）而无法支持 S256，并且通过带外配置或授权服务器元数据 [RFC8414] 知道服务器支持 plain，则允许客户端仅使用 plain。

code_challenge 的 ABNF 如下：

```
code-challenge = 43*128unreserved
unreserved = ALPHA / DIGIT / "-" / "." / "_" / "~"
ALPHA = %x41-5A / %x61-7A
DIGIT = %x30-39
```

code_challenge 和 code_verifier 属性取自 OAuth 2.0 扩展"代码交换的证明密钥"（PKCE）[RFC7636]，该技术最初是在那里开发的。

授权服务器必须支持 code_challenge 和 code_verifier 参数。

客户端必须使用 code_challenge 和 code_verifier，除第 7.5.1 节中描述的条件外，授权服务器必须强制执行其使用。即使在这种情况下，仍建议使用和强制执行上述 code_challenge 和 code_verifier。

state 和 scope 参数不应包含明文形式的敏感客户端或资源所有者信息，因为它们可能通过不安全通道传输或安全存储。

客户端使用 HTTP 重定向或其他用户代理可用的方式将资源所有者引导到构建的 URI。

例如，客户端引导用户代理发出以下 HTTPS 请求（仅为显示目的添加额外换行）：

```
GET /authorize?response_type=code&client_id=s6BhdRkqt3&state=xyz
    &redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb
    &code_challenge=6fdkQaPm51l13DSukcAH3Mdx7_ntecHYd1vi3n0hMZY
    &code_challenge_method=S256 HTTP/1.1
Host: server.example.com
```

授权服务器验证请求以确保所有必需的参数都存在且有效。

特别是，授权服务器必须验证请求中的 redirect_uri（如果存在），确保它与客户端注册期间预先建立（Section 2）的一个已注册重定向 URI 匹配。在比较两个 URI 时，授权服务器必须确保两个 URI 相等，详见 [RFC3986] 第 6.2.1 节简单字符串比较。唯一的例外是使用 localhost URI 的原生应用程序：在这种情况下，授权服务器必须允许可变的端口号，如 [RFC8252] 第 7.3 节中所述。

如果请求有效，授权服务器对资源所有者进行认证并获得授权决定（通过询问资源所有者或通过其他方式建立批准）。

当建立决定后，授权服务器使用 HTTP 重定向响应或用户代理可用的其他方式将用户代理引导到提供的客户端重定向 URI。

### 4.1.2. 授权响应

如果资源所有者授予访问请求，授权服务器颁发授权码，并通过将以下参数添加到重定向 URI 的查询组件（使用附录 C.1 中描述的查询字符串序列化）将其传递给客户端，除非扩展另有指定：

**"code"**：必需的。授权码由授权服务器生成，对客户端是不透明的。授权码必须在颁发后不久过期，以减少泄露风险。建议最大授权码生命周期为 10 分钟。授权码绑定到客户端标识符、代码挑战和重定向 URI。

**"state"**：如果客户端授权请求中存在 state 参数，则是必需的。完全相同地接收自客户端的值。

**"iss"**：可选的。客户端可用于防止混消攻击的授权服务器标识符，如果客户端与多个授权服务器交互。详见第 7.14 节和 [RFC9207]，了解此参数何时必要，以及客户端如何使用它来防止混消攻击。

例如，授权服务器通过发送以下 HTTP 响应将用户代理重定向：

```
HTTP/1.1 302 Found
Location: https://client.example.com/cb?code=SplxlOBeZQQYbYS6WxSbIA
          &state=xyz&iss=https%3A%2F%2Fauthorization-server.example.com
```

客户端必须忽略无法识别的响应参数。授权码字符串大小由本规范未定义。客户端应避免对代码值大小做出假设。授权服务器应记录其颁发的任何值的大小。

授权服务器必须将 code_challenge 和 code_challenge_method 值与颁发的授权码关联，以便以后可以验证代码挑战。

服务器用于将 code_challenge 与颁发的代码关联的确切方法不在本规范范围内。代码挑战可以存储在服务器上并在那里与代码关联。code_challenge 和 code_challenge_method 值可以以加密形式存储在代码本身中，但服务器不得在响应参数中以 AS 以外的实体可以提取的形式包含 code_challenge 值。

客户端必须防止攻击者向授权响应注入（重放）授权码。使用 code_challenge 和 code_verifier 可以防止授权码注入，因为授权服务器将拒绝 code_verifier 不匹配的令牌请求。详见第 7.5.1 节。

#### 4.1.2.1. 授权错误响应

如果请求因缺失、无效或不匹配的重定向 URI 而失败，或者客户端标识符缺失或无效，授权服务器不得将用户代理重定向到无效的重定向 URI，并应告知资源所有者错误，例如通过在浏览器中显示消息。

授权服务器必须拒绝来自公开客户端的没有 code_challenge 的请求，并且必须拒绝来自其他客户端的此类请求，除非有合理的保证表明客户端以其他方式缓解了授权码注入。详见第 7.5.1 节。

如果服务器不支持请求的 code_challenge_method 转换，授权端点必须返回错误值为 invalid_request 的授权错误响应。error_description 或 error_uri 的响应应解释错误的性质，例如，不支持转换算法。

如果资源所有者拒绝访问请求或请求因除缺失或无效重定向 URI 以外的原因而失败，授权服务器通过将用户代理重定向到重定向 URI 并将以下参数添加到重定向 URI 的查询组件（如附录 C.1 中所述）来通知客户端：

**"error"**：必需的。来自以下情况的单个 ASCII [USASCII] 错误代码：

* **"invalid_request"**：请求缺少必需的参数、包含无效参数值、多次包含参数，或格式不正确。

* **"unauthorized_client"**：客户端无权使用此方法请求授权码。

* **"access_denied"**：资源所有者或授权服务器拒绝了请求。

* **"unsupported_response_type"**：授权服务器不支持使用此方法获取授权码。

* **"invalid_scope"**：请求的作用域无效、未知或格式错误。

* **"server_error"**：授权服务器遇到阻止其完成请求的意外状况。（需要此错误代码，因为无法通过 HTTP 重定向向客户端返回 500 Internal Server Error HTTP 状态码。）

* **"temporarily_unavailable"**：授权服务器当前由于服务器临时过载或维护而无法处理请求。（需要此错误代码，因为无法通过 HTTP 重定向向客户端返回 503 Service Unavailable HTTP 状态码。）

错误参数的值不得包含 %x20-21 / %x23-5B / %x5D-7E 集合之外的字符。

**"error_description"**：可选的。提供附加信息的可读性 ASCII [USASCII] 文本，用于帮助客户端开发者理解发生的错误。error_description 参数的值不得包含 %x20-21 / %x23-5B / %x5D-7E 集合之外的字符。

**"error_uri"**：可选的。一个 URI，标识包含错误信息的可读性网页，用于向客户端开发者提供有关错误的附加信息。error_uri 参数的值必须符合 URI-reference 语法，因此不得包含 %x21 / %x23-5B / %x5D-7E 集合之外的字符。

**"state"**：如果客户端授权请求中存在 state 参数，则是必需的。完全相同地接收自客户端的值。

**"iss"**：可选的。授权服务器的标识符。详见上面的第 4.1.2 节。

例如，授权服务器通过以下 HTTP 响应重定向用户代理来指示请求被拒绝：

```
HTTP/1.1 302 Found
Location: https://client.example.com/cb?error=access_denied
          &state=xyz&iss=https%3A%2F%2Fauthorization-server.example.com
```

### 4.1.3. 令牌端点扩展

授权许可类型在令牌端点通过 grant_type 值 authorization_code 来标识。

如果设置了此值，则支持第 3.2.2 节之外的以下附加令牌请求参数：

**"code"**：必需的。从授权服务器收到的授权码。

**"code_verifier"**：如果授权请求中包含了 code_challenge 参数，则是必需的。否则不得使用。原始代码验证器字符串。

**"client_id"**：如果客户端未按照第 3.2.1 节中的描述进行认证，则是必需的。

对于给定的授权码，授权服务器必须只返回一次访问令牌。

如果使用与先前成功的令牌请求相同的授权码发出第二次有效令牌请求，授权服务器必须拒绝请求，并应撤销（如果可能）先前基于该授权码颁发的所有访问令牌和刷新令牌。详见第 7.5.2 节。

例如，客户端发出以下 HTTPS 请求（仅为显示目的添加额外换行）：

```
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=SplxlOBeZQQYbYS6WxSbIA
&code_verifier=3641a2d12d66101249cdf7a79c000c1f8c05d2aafcf14bf146497bed
```

除了第 3.2.2 节中的处理规则外，授权服务器还必须：

* 确保授权码颁发给了已认证的机密客户端，或者如果客户端是公开的，确保代码颁发给了请求中的 client_id

* 验证授权码有效

* 验证 code_verifier 参数当且仅当授权请求中存在 code_challenge 参数时才存在

* 如果存在 code_verifier，通过首先根据客户端指定的 code_challenge_method 方法对其进行转换，从收到的 code_verifier 计算代码挑战，并将其与先前关联的 code_challenge 进行比较，从而验证 code_verifier

* 如果令牌请求中关联的授权请求中没有 code_challenge，授权服务器必须拒绝令牌请求。

有关关于令牌请求中 redirect_uri 参数与 OAuth 2.0 客户端向后兼容性的详细信息，请参见第 10.2 节。

## 4.2. 客户端凭证许可

当客户端请求访问其控制下的受保护资源时，或请求访问先前与授权服务器约定的另一个资源所有者的受保护资源时（其方法超出本规范范围），客户端可以使用其客户端凭证（或其他支持的认证方式）请求访问令牌。

客户端凭证许可类型只能由机密客户端使用。

```
     +---------+                                  +---------------+
     |         |                                  |               |
     |         |>--(1)- Client Authentication --->| Authorization |
     | Client  |                                  |     Server    |
     |         |<--(2)---- Access Token ---------<|               |
     |         |                                  |               |
     +---------+                                  +---------------+

                Figure 4: Client Credentials Grant
```

图 4 中说明的客户端凭证许可的使用包括以下步骤：

(1) 客户端向授权服务器进行认证，并从令牌端点请求访问令牌。

(2) 授权服务器对客户端进行认证，如果有效，则颁发访问令牌。

### 4.2.1. 令牌端点扩展

客户端凭证许可类型在令牌端点通过 grant_type 值 client_credentials 来标识。

如果设置了此值，则支持第 3.2.2 节之外的以下附加令牌请求参数：

**"scope"**：可选的。访问请求的作用域，如第 1.4.1 节所述。

例如，客户端使用传输层安全发出以下 HTTP 请求（仅为显示目的添加额外换行）：

```
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
```

授权服务器必须对客户端进行认证。

## 4.3. 刷新令牌许可

刷新令牌是授权服务器颁发给客户端的凭证，可用于根据现有许可获取新的（新鲜的）访问令牌。客户端使用此选项是因为之前的访问令牌已过期，或者客户端之前获取的访问令牌的作用域比相应许可批准的作用域更窄，后来需要在同一许可下请求具有不同作用域的访问令牌。

刷新令牌在传输和存储过程中必须保持机密，仅在授权服务器和颁发给的客户端之间共享。授权服务器必须维护刷新令牌与颁发给的客户端之间的绑定。

每当可以认证客户端身份时，授权服务器必须验证刷新令牌与客户端身份之间的绑定。当无法进行客户端认证时，授权服务器应颁发发送者约束的刷新令牌或使用第 4.3.1 节中描述的刷新令牌轮换。

授权服务器必须确保刷新令牌不能被未经授权的方生成、修改或猜测以产生有效刷新令牌。

### 4.3.1. 令牌端点扩展

刷新令牌许可类型在令牌端点通过 grant_type 值 refresh_token 来标识。

如果设置了此值，则支持第 3.2.2 节之外的以下附加参数：

**"refresh_token"**：必需的。颁发给客户端的刷新令牌。

**"scope"**：可选的。访问请求的作用域，如第 1.4.1 节所述。请求的作用域不得包括资源所有者最初未授予的任何作用域，如果省略，则被视为等于资源所有者最初授予的作用域。

由于刷新令牌通常是用于请求额外访问令牌的长期凭证，刷新令牌绑定到颁发给的客户端。机密客户端必须按照第 3.2.1 节中的描述进行认证。

例如，客户端使用传输层安全发出以下 HTTP 请求（仅为显示目的添加额外换行）：

```
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
```

除了第 3.2.2 节中的处理规则外，授权服务器还必须：

* 如果请求中包含客户端认证，确保刷新令牌颁发给了已认证的客户端，或者如果请求中包含 client_id，确保刷新令牌颁发给了匹配的客户端

* 验证此刷新令牌对应的许可仍然有效

* 验证刷新令牌

授权服务器必须使用以下方法之一来检测公开客户端的刷新令牌重放：

* **发送者约束的刷新令牌：** 授权服务器使用加密方式将刷新令牌绑定到某个客户端实例，例如使用 DPoP [RFC9449] 或 mTLS [RFC8705]。

* **刷新令牌轮换：** 授权服务器在每次访问令牌刷新响应时颁发新的刷新令牌。先前的刷新令牌被无效化，但授权服务器保留关系信息。如果刷新令牌被泄露并随后被攻击者和合法客户端使用，其中一个将出示无效化的刷新令牌，这将通知授权服务器发生了泄露。授权服务器无法确定哪个提交方提交了无效的刷新令牌，但它将撤销活动的刷新令牌及其关联的访问授权许可。这以迫使合法客户端获取新的授权许可为代价阻止了攻击。

实现注意：刷新令牌所属的许可可以编码到刷新令牌本身中。这可以使授权服务器有效地确定刷新令牌所属的许可，进而确定需要撤销的所有刷新令牌。授权服务器必须确保在这种情况下刷新令牌值的完整性，例如使用签名。

### 4.3.2. 刷新令牌响应

如果有效且已授权，授权服务器按照第 3.2.3 节中的描述颁发访问令牌。

授权服务器可以颁发新的刷新令牌，在这种情况下，客户端必须丢弃旧的刷新令牌并用新的刷新令牌替换。

### 4.3.3. 刷新令牌建议

授权服务器可以在向客户端颁发新刷新令牌后撤销旧的刷新令牌。如果颁发新刷新令牌，刷新令牌作用域必须与客户端请求中包含的刷新令牌的作用域相同。

如果发生安全事件，授权服务器可以自动撤销刷新令牌，例如：

* 密码更改

* 在授权服务器注销

如果客户端处于非活动状态一段时间（即刷新令牌一段时间未用于获取新访问令牌），刷新令牌应该过期。过期时间由授权服务器自行决定。它可能是全局值，也可能是基于客户端策略或与刷新令牌关联的许可（及其敏感性）确定的。

## 4.4. 扩展许可

客户端通过使用绝对 URI（由授权服务器定义）作为令牌端点的 grant_type 参数的值，并添加任何必要的额外参数来使用扩展许可类型。

例如，要在用户已在单独设备上授权客户端后，使用 [RFC8628] 定义的设备授权许可请求访问令牌，客户端发出以下 HTTPS 请求（仅为显示目的添加额外换行）：

```
  POST /token HTTP/1.1
  Host: server.example.com
  Content-Type: application/x-www-form-urlencoded

  grant_type=urn%3Aietf%3Aparams%3Aoauth%3Agrant-type%3Adevice_code
  &device_code=GmRhmhcxhwEzkoEqiMEg_DnyEysNkuNhszIySk9eS
  &client_id=C409020731
```

如果访问令牌请求有效且已授权，授权服务器按照第 3.2.3 节中的描述颁发访问令牌和可选的刷新令牌。如果请求的客户端认证失败或无效，授权服务器返回第 3.2.4 节中描述的错误响应。
