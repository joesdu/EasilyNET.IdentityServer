# 10. 与 OAuth 2.0 的差异

本草案整合了 OAuth 2.0 [RFC6749]、OAuth 2.0 原生应用程序 [RFC8252]、代码交换证明密钥 [RFC7636]、基于浏览器的应用程序 OAuth 2.0 [I-D.ietf-oauth-browser-based-apps]、OAuth 安全最佳当前实践 [RFC9700] 和持有者令牌使用 [RFC6750] 中的功能。

如果后续草案更新或取代了原始 [RFC6749] 中的功能，则本草案中的该功能会使用后续草案中描述的规范性更改进行更新，或完全删除。

下面列出了与 OAuth 2.0 的差异的非规范性列表：

* 授权码许可扩展了 PKCE [RFC7636] 的功能，使得根据本规范使用授权码许可的默认方法需要添加 PKCE 参数

* 重定向 URI 必须使用与 [RFC9700] 第 4.1.3 节一致的精确字符串匹配进行比较

* 隐式许可（response_type=token）已从本规范中省略，如 [RFC9700] 第 2.1.2 节所述

* 资源所有者密码凭证许可已从本规范中省略，如 [RFC9700] 第 2.4 节所述

* 持有者令牌使用省略了在 URI 查询字符串中使用持有者令牌，如 [RFC9700] 第 4.3.2 节所述

* 公开客户端的刷新令牌必须要么是发送者约束的，要么是一次性的，如 [RFC9700] 第 4.14.2 节所述

* 令牌请求中包含授权码的请求不再包含 redirect_uri 参数

* 授权服务器必须在请求正文中支持客户端凭证

## 10.1. 删除 OAuth 2.0 隐式许可

OAuth 2.0 隐式许可已从 OAuth 2.1 中省略，因为它已在 [RFC9700] 中弃用。

删除隐式许可的目的是不再在授权响应中颁发访问令牌，因为此类令牌容易泄露和注入，并且无法被约束为特定客户端。客户端使用 response_type=token 参数表示此行为。response_type 参数的此值在 OAuth 2.1 中不再定义。

删除 response_type=token 不影响从授权端点返回其他伪凭证的其他扩展响应类型，例如 [OpenID.Connect] 定义的 response_type=id_token。

## 10.2. 令牌请求中的重定向 URI 参数

在 OAuth 2.0 中，授权码流程（第 [RFC6749] 第 4.1.3 节）中令牌端点的请求包含可选的 redirect_uri 参数。该参数旨在防止授权码注入攻击，如果原始授权请求中发送了 redirect_uri 参数，则需要该参数。授权请求仅在向特定客户端注册了多个重定向 URI 时才需要 redirect_uri 参数。然而，实际上，许多授权服务器实现要求即使只注册了一个也要求在授权请求中包含 redirect_uri 参数，导致 redirect_uri 参数也需要在令牌端点。

在 OAuth 2.1 中，授权码注入通过 code_challenge 和 code_verifier 参数防止，使 redirect_uri 参数在令牌请求中的包含毫无用处。因此，它已被删除。

为了支持同时希望支持 OAuth 2.0 和 OAuth 2.1 客户端的授权服务器的向后兼容性，授权服务器必须允许客户端在令牌请求（第 4.1.3 节）中发送 redirect_uri 参数，并必须按照 [RFC6749] 中所述强制执行该参数。授权服务器可以使用请求中的 client_id 来确定是否为知道将使用旧 OAuth 2.0 行为的特定客户端强制执行此行为。

仅遵循 OAuth 2.1 建议的客户端不会在令牌请求中发送 redirect_uri，因此不会与期望令牌请求中包含该参数的授权服务器兼容。
