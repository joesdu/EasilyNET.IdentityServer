# 6. 扩展性

## 6.1. 定义访问令牌类型

访问令牌类型可以通过以下两种方式之一定义：在访问令牌类型注册表中注册（遵循 [RFC6749] 第 11.1 节中的程序），或使用唯一的绝对 URI 作为其名称。

### 6.1.1. 已注册的访问令牌类型

[RFC6750] 在 [RFC6749] 第 11.4 节中建立了一个通用注册表，用于在 OAuth 令牌认证方案之间共享错误值。

主要为 OAuth 令牌认证设计的新认证方案应定义一种向客户端提供错误状态码的机制，其中允许的错误值在本规范建立的错误注册表中注册。

此类方案可以将有效错误代码限制为注册值的子集。如果错误代码使用命名参数返回，参数名称应为 error。

能够用于 OAuth 令牌认证但并非主要为此目的设计的其他方案可以同样将其错误值绑定到注册表。

新认证方案可以选择也指定使用 error_description 和 error_uri 参数以与本规范中平行使用的方式返回错误信息。

类型名称必须符合 type-name ABNF。如果类型定义包括新的 HTTP 认证方案，类型名称应与 HTTP 认证方案名称相同（如 [RFC2617] 中定义）。示例令牌类型保留用于示例。

```
type-name  = 1*name-char
name-char  = "-" / "." / "_" / DIGIT / ALPHA
```

### 6.1.2. 供应商特定的访问令牌类型

利用 URI 名称的类型应限于供应商特定的实现，这些实现不普遍适用，而是特定于使用它们的资源服务器的实现细节。

所有其他类型必须注册。

## 6.2. 定义新的端点参数

与授权端点或令牌端点一起使用的新请求或响应参数按照 [RFC6749] 第 11.2 节中的程序在 OAuth 参数注册表中定义和注册。

参数名称必须符合 param-name ABNF，参数值语法必须定义明确（例如，使用 ABNF 或对现有参数语法的引用）。

```
param-name  = 1*name-char
name-char   = "-" / "." / "_" / DIGIT / ALPHA
```

不普遍适用且特定于使用它们的授权服务器实现细节的未注册供应商特定参数扩展应利用不太可能与其他注册值冲突的供应商特定前缀（例如，以 'companyname_' 开头）。

## 6.3. 定义新的授权许可类型

可以通过为 grant_type 参数分配唯一的绝对 URI 来定义新的授权许可类型。如果扩展许可类型需要额外的令牌端点参数，必须按照 [RFC6749] 第 11.2 节中的描述在 OAuth 参数注册表中注册。

## 6.4. 定义新的授权端点响应类型

与授权端点一起使用的新响应类型按照 [RFC6749] 第 11.3 节中的程序在授权端点响应类型注册表中定义和注册。响应类型名称必须符合 response-type ABNF。

```
response-type  = response-name *( SP response-name )
response-name  = 1*response-char
response-char  = "_" / DIGIT / ALPHA
```

如果响应类型包含一个或多个空格字符（%x20），则作为空格分隔的值列表进行比较，其中值的顺序不重要。只能注册一种值顺序，覆盖相同值集的所有其他排列。

例如，扩展可以定义和注册 code other_token 响应类型。注册后，相同的组合不能注册为 other_token code，但两个值都可用于表示相同的响应类型。

## 6.5. 定义附加错误代码

在协议扩展（即访问令牌类型、扩展参数或扩展许可类型）需要额外的错误代码与授权码许可错误响应（第 4.1.2.1 节）、令牌错误响应（第 3.2.4 节）或资源访问错误响应（第 5.3 节）一起使用的情况下，可以定义此类错误代码。

如果它们所使用的扩展是已注册的访问令牌类型、已注册的端点参数或扩展许可类型，则扩展错误代码必须注册（遵循 [RFC6749] 第 11.4 节中的程序）。与未注册扩展一起使用的错误代码可以注册。

错误代码必须符合 error ABNF，并应在可能时使用标识名称作为前缀。例如，为扩展参数 example 设置无效值的错误应命名为 example_invalid。

```
error      = 1*error-char
error-char = %x20-21 / %x23-5B / %x5D-7E
```
