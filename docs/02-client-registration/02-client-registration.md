# 2. 客户端注册

在启动协议之前，客户端必须在授权服务器上建立一个标识符（第 2.2 节）。客户端标识符与授权服务器建立关联的方式不在本规范范围内，但通常涉及客户端开发者在授权服务器的网站上手动注册客户端（创建账户并同意服务条款之后），或使用动态客户端注册 [RFC7591]。扩展也可以定义其他以编程方式建立客户端注册的方法。

客户端注册不需要客户端与授权服务器之间的直接交互。当授权服务器支持时，注册可以依赖其他方式来建立信任并获取所需的客户端属性（例如重定向 URI、客户端类型）。例如，注册可以使用自颁发或第三方颁发的断言来完成，或由授权服务器使用可信通道执行客户端发现。

客户端注册必须包括：

* 第 2.1 节中描述的客户端类型

* 所使用的许可类型所需的客户端详细信息，例如第 2.3 节中描述的重定向 URI

* 授权服务器要求的任何其他信息（例如应用程序名称、网站、描述、徽标图片、法律条款的接受）

动态客户端注册 [RFC7591] 定义了一个通用的客户端数据模型，即使在手动客户端注册时也可以使用。

## 2.1. 客户端类型

OAuth 2.1 根据客户端能否安全地与授权服务器进行认证，定义了两种客户端类型：

**"机密客户端"（confidential）**：拥有 AS 凭证的客户端被指定为"机密客户端"

**"公开客户端"（public）**：没有凭证的客户端称为"公开客户端"

任何拥有凭证的客户端都必须采取预防措施，防止其凭证泄露和滥用。

客户端认证允许授权服务器确保其在 OAuth 流程中与某个特定客户端（由其 client_id 标识）交互。授权服务器可能根据其对正在与合法客户端通信的置信度做出策略决策，例如是否每次授权时都提示用户同意，还是仅在第一次时提示。

授权服务器是否以及如何验证客户端或其提供/运营该客户端的一方的身份不在本规范范围内。授权服务器在决定是否允许客户端访问更敏感的资源和服务（如客户端凭证许可类型）以及多久提示用户一次同意时，应考虑对客户端身份的置信度水平。

不要求授权服务器支持特定的客户端类型。

单个 client_id 不应被当作多种类型的客户端处理。

本规范围绕以下客户端配置文件设计：

**"Web 应用程序"（web application）**：Web 应用程序是在 Web 服务器上运行的客户端。资源所有者通过资源所有者使用的设备上的用户代理呈现的 HTML 用户界面访问客户端。颁发给客户端的客户端凭证以及任何访问令牌存储在 Web 服务器上，不会暴露给资源所有者或被资源所有者访问。

**"基于浏览器的应用程序"（browser-based application）**：基于浏览器的应用程序是一种客户端，其客户端代码从 Web 服务器下载并在资源所有者使用的设备上的用户代理（例如 Web 浏览器）中执行。协议数据和凭证很容易被资源所有者访问（通常可见）。如果此类应用程序希望使用客户端凭证，建议利用后端模式。由于此类应用程序驻留在用户代理中，它们在请求授权时可以无缝地利用用户代理的能力。

**"原生应用程序"（native application）**：原生应用程序是安装在资源所有者使用的设备上并执行的客户端。协议数据和凭证可被资源所有者访问。假设应用程序中包含的任何客户端认证凭证都可以被提取。动态颁发的访问令牌和刷新令牌可以获得可接受的保护级别。在某些平台上，这些凭证受到保护，免受同一设备上其他应用程序的侵害。如果此类应用程序希望使用客户端凭证，建议利用后端模式，或使用动态客户端注册 [RFC7591] 在运行时颁发凭证。

## 2.2. 客户端标识符

每个客户端在授权服务器的上下文中由客户端标识符标识——一个唯一字符串，表示客户端提供的注册信息。虽然授权服务器通常自己颁发客户端标识符，但它也可以为客户端标识符由授权服务器以外的方创建的客户端提供服务。客户端标识符不是秘密；它会暴露给资源所有者，不得单独用于客户端认证。客户端标识符在授权服务器的上下文中是唯一的。

客户端标识符是一个不透明的字符串，其大小由本规范未定义。客户端应避免对标识符大小做出假设。授权服务器应记录其颁发的任何标识符的大小。

如果授权服务器支持由授权服务器以外的方颁发客户端标识符的客户端，授权服务器应采取预防措施，避免客户端冒充资源所有者（如第 7.4 节中所述）。

## 2.3. 客户端重定向端点

客户端重定向端点（也称为"重定向端点"）是授权服务器在完成与资源所有者的交互后将用户代理重定向回的客户端的 URI。

授权服务器将用户代理重定向到客户端在客户端注册过程中预先与授权服务器建立的重定向端点之一。

重定向 URI 必须是 [RFC3986] 第 4.3 节定义的绝对 URI。重定向 URI 可以包含查询字符串组件（附录 C.1），在添加额外查询参数时必须保留。重定向 URI 不得包含片段组件。

### 2.3.1. 注册要求

授权服务器必须要求客户端注册其完整的重定向 URI（包括路径组件）。授权服务器必须拒绝指定未注册的重定向 URI 的授权请求，但回环重定向除外，回环重定向需要精确匹配，但端口 URI 组件除外，详见第 4.1.1 节。

授权服务器可以允许客户端注册多个重定向 URI。

注册可以发生在带外，例如在授权服务器上配置客户端信息的手动步骤，或者可以在运行时发生，例如推送授权请求 [RFC9126] 中的初始 POST。

对于基于私有 URI 方案的重定向 URI，授权服务器应强制执行第 8.4.3 节中的要求，即客户端使用基于反向域名的方案。至少，任何不包含句点字符（.）的私有 URI 方案都应该被拒绝。

除了抗碰撞特性之外，这可以帮助在发生争议时证明所有权，当两个应用程序声称使用相同的私有 URI 方案时（其中一个应用程序表现恶意）。例如，如果两个应用程序都声称使用 com.example.app，example.com 的所有者可以向应用程序商店运营商请求删除假冒应用程序。如果使用通用 URI 方案，这种请求更难证明。

客户端不得暴露将用户的浏览器转发到从查询参数获取的任意 URI 的 URL（"开放重定向器"），如第 7.12 节所述。开放重定向器可能导致授权码和访问令牌泄露。

如果需要，客户端可以使用 state 请求参数来实现每次请求的自定义，而不是每个请求更改重定向 URI。

如果不要求注册重定向 URI，攻击者可以将授权端点用作开放重定向器，如第 7.12 节所述。

### 2.3.2. 多个重定向 URI

如果为客户端注册了多个重定向 URI，客户端必须在授权请求中使用 redirect_uri 请求参数（第 4.1.1 节）包含一个重定向 URI。如果只为客户端注册了一个重定向 URI，则 redirect_uri 请求参数是可选的。

### 2.3.3. 防止 CSRF 攻击

客户端必须防止跨站请求伪造（CSRF）攻击。在此上下文中，CSRF 是指指向重定向端点的请求，这些请求不是来自授权服务器，而是来自恶意第三方（详见 [RFC6819] 第 4.4.1.8 节）。如果客户端已确保授权服务器支持 code_challenge 参数，则可以依赖该机制提供的 CSRF 保护。在 OpenID Connect 流程中，验证 nonce 参数可提供 CSRF 保护。否则，必须使用绑定到用户代理的安全性较强的一次性 CSRF 令牌（通过 state 参数携带）进行 CSRF 保护（见第 7.9 节）。

### 2.3.4. 防止混消攻击

当 OAuth 客户端只能与一个授权服务器交互时，不需要混消防御。但是，在 OAuth 客户端与两个或多个授权服务器交互的场景中，客户端必须防止混消攻击。为了防止混消攻击，客户端必须仅处理来自它发送相应请求的颁发者的重定向响应，并且来自发起此授权请求的同一用户代理。

有关防止混消攻击的两种不同防御措施的详细说明，请参见第 7.14 节。

### 2.3.5. 无效端点

如果授权请求因缺失、无效或不匹配的重定向 URI 而验证失败，授权服务器应告知资源所有者错误，并且不得自动将用户代理重定向到无效的重定向 URI。

### 2.3.6. 端点内容

对客户端端点的重定向请求通常会导致 HTML 文档响应，由用户代理处理。如果 HTML 响应直接作为重定向请求的结果返回，则包含在 HTML 文档中的任何脚本都将完全访问重定向 URI 及其包含的伪凭证（如授权码）。此外，包含授权码的请求 URL 可能会在 HTTP Referer 头中发送到页面中加载的任何嵌入式图像、样式表和其他元素。

客户端不应在重定向 URI 端点响应中包含任何第三方脚本（例如第三方分析、社交插件、广告网络）。相反，它应该从 URI 中提取伪凭证，并将用户代理再次重定向到另一个端点，而不暴露伪凭证（在 URI 或其他地方）。如果包含第三方脚本，客户端必须确保自己的脚本（用于从 URI 中提取和移除凭证）首先执行。

## 2.4. 客户端认证

只有当颁发/注册和分发基础凭证的过程确保其机密性时，授权服务器才能仅依赖客户端认证。

对于机密客户端，授权服务器可以接受满足其安全要求的任何形式的客户端认证（例如客户端密钥、公钥/私钥对）。

建议使用非对称（基于公钥）方法进行客户端认证，例如 mTLS [RFC8705] 或使用签名 JWT（"私钥 JWT"），符合 [RFC7521]、[RFC7523] 及其更新 [I-D.ietf-oauth-rfc7523bis]（在 [OpenID.Connect] 中定义为客户端认证方法 private_key_jwt）。当使用此类客户端认证方法时，授权服务器不需要存储敏感的对称密钥，使这些方法更能抵抗多种攻击，并使客户端能够管理自己的密钥和密钥轮换。

当使用基于 JWT 的客户端认证时，客户端和授权服务器必须遵循 [I-D.ietf-oauth-rfc7523bis] 中关于 aud 值的更新指导。

当客户端认证不可能时，授权服务器应采用其他方式验证客户端身份——例如，要求注册客户端重定向 URI 或请资源所有者确认身份。当请求资源所有者授权时，有效的重定向 URI 不足以验证客户端身份，但可用于防止在获得资源所有者授权后将凭证传递给假冒客户端。

客户端不得在每个请求中使用多种认证方法，以防止哪个认证机制是请求的权威来源发生冲突。

授权服务器必须考虑与未认证客户端交互的安全隐患，并采取措施限制向此类客户端颁发的令牌的潜在暴露（例如限制刷新令牌的生命周期）。

授权服务器将某个客户端身份关联的特权必须取决于对客户端身份识别和客户端凭证生命周期管理的整体过程评估。详见第 7.2 节。

### 2.4.1. 客户端密钥

为了支持持有客户端密钥的机密客户端，授权服务器必须支持客户端使用以下参数在请求内容中包含客户端凭证：

**"client_id"**：必需的。 在第 2.2 节描述的注册过程中颁发给客户端的客户端标识符。

**"client_secret"**：必需的。 客户端密钥。

参数只能通过请求内容传输，不得包含在请求 URI 中。

这也称为 [RFC7591] 第 2 节中定义的 client_secret_post。

例如，使用内容参数刷新访问令牌的请求（第 4.3 节）（仅为显示目的添加额外换行）：

```
POST /token HTTP/1.1
Host: server.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
&client_id=s6BhdRkqt3&client_secret=7Fjfp0ZBr1KtDRbnfVdmIw
```

授权服务器可以支持 HTTP Basic 认证方案来认证已颁发客户端密钥的客户端。

当使用 [RFC9110] 第 11 节中定义的 HTTP Basic 认证方案向授权服务器进行认证时，客户端标识符使用应用程序/x-www-form-urlencoded 编码算法（per Appendix B）进行编码，并将编码值用作用户名；客户端密钥使用相同的算法进行编码并用作密码。

这也称为 [RFC7591] 第 2 节中定义的 client_secret_basic。

例如（仅为显示目的添加额外换行）：

```
Authorization: Basic czZCaGRSa3F0Mzo3RmpmcDBaQnIxS3REUmJuZlZkbUl3
```

注意：这种方法——首先对客户端标识符和密钥进行表单编码，然后使用编码值作为 HTTP Basic 认证的用户名和密码——在过去导致了许多互操作性问题。一些实现错过了编码步骤，或者决定只对某些字符进行编码，或者在验证凭证时忽略了编码要求，导致客户端必须特殊处理如何向各个授权服务器呈现凭证。将凭证包含在请求内容中可以避免编码问题，从而产生更具互操作性的实现。

由于客户端密钥认证方法涉及密码，授权服务器必须保护任何使用它的端点免受暴力攻击。

### 2.4.2. 其他认证方法

授权服务器可以支持任何满足其安全要求的合适认证方案。当使用其他认证方法时，授权服务器必须定义客户端标识符（注册记录）与认证方案之间的映射。

一些额外的认证方法（如 mTLS [RFC8705] 和私钥 JWT（[RFC7523]、[I-D.ietf-oauth-rfc7523bis]））在"OAuth 令牌端点认证方法（https://www.iana.org/assignments/oauth-parameters/oauth-parameters.xhtml#token-endpoint-auth-method）"注册表中定义，除了保护令牌端点的特定用途外，还可能作为通用客户端认证方法有用。

## 2.5. 未注册客户端

本规范不要求客户端在授权服务器注册。但是，使用未注册客户端超出了本规范的范围，需要额外的安全分析和互操作性影响审查。
