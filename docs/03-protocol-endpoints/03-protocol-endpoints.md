# 3. 协议端点

授权过程利用两个授权服务器端点（HTTP 资源）：

* **授权端点** - 用于客户端通过用户代理重定向获取资源所有者的授权

* **令牌端点** - 用于客户端交换授权许可以获取访问令牌，通常带有客户端认证

以及一个客户端端点：

* **重定向端点** - 用于授权服务器通过资源所有者用户代理将包含授权凭证的响应返回给客户端

并非每种授权许可类型都使用两个端点。扩展许可类型可以根据需要定义额外的端点。

## 3.1. 授权端点

授权端点用于与资源所有者交互并获取授权许可。授权服务器必须首先对资源所有者进行认证。授权服务器对资源所有者进行认证的方式（例如用户名和密码登录、通行密钥、联合登录或使用已建立的会话）超出了本规范的范围。

客户端获取授权端点 URL 的方式超出了本规范的范围，但 URL 通常在服务文档中提供，或在授权服务器的元数据文档 [RFC8414] 中提供。

授权端点 URL 不得包含片段组件，但可以包含查询字符串组件（附录 C.1），在添加额外查询参数时必须保留。

授权服务器必须支持在授权端点上使用 HTTP GET 方法 [RFC9110 第 9.3.1 节]，也可以支持 POST 方法（[RFC9110] 第 9.3.3 节）。

授权服务器必须忽略发送到授权端点的无法识别的请求参数。

本规范定义的请求和响应参数不得包含多次。此要求也适用于扩展定义的参数，除非扩展明确为特定参数定义否则。发送时没有值的参数必须被视为从请求中省略。

重定向可能包含用户凭证的请求的授权服务器必须避免意外转发这些用户凭证（详见第 7.5.3 节）。

授权端点不得支持跨域资源共享 [WHATWG.CORS]，因为客户端不直接访问此端点，而是将用户代理重定向到该端点。

## 3.2. 令牌端点

令牌端点用于客户端使用诸如第 4 节和第 4.3 节中描述的许可来获取访问令牌。

客户端获取令牌端点 URL 的方式超出了本规范的范围，但 URL 通常在服务文档中提供，并在客户端开发期间进行配置，或在授权服务器的元数据文档 [RFC8414] 中提供并在运行时以编程方式获取。

令牌端点 URL 不得包含片段组件，但可以包含查询字符串组件（附录 C.1）。

客户端在向令牌端点发出请求时必须使用 HTTP POST 方法。

授权服务器必须忽略发送到令牌端点的无法识别的请求参数。

发送时没有值的参数必须被视为从请求中省略。本规范定义的请求和响应参数不得包含多次。此要求也适用于扩展定义的参数，除非扩展明确为特定参数定义否则。

希望支持基于浏览器的应用程序（例如，专门在客户端 JavaScript 中运行且无法访问支持性后端服务器的应用程序）的授权服务器需要确保令牌端点支持必要的 CORS [WHATWG.CORS] 头，以允许应用程序查看响应。如果授权服务器向应用程序提供其他端点（如元数据 URL、动态客户端注册、撤销、introspection、发现或用户信息端点），这些端点也可能被基于浏览器的应用程序访问，也需要定义 CORS 头以允许访问。详见 [I-D.ietf-oauth-browser-based-apps]。

### 3.2.1. 客户端认证

机密客户端在向令牌端点发出请求时必须按照第 2.4 节中的描述进行认证。

客户端认证用于：

* 强制将刷新令牌和授权码绑定到颁发给的客户端。当授权码通过不安全通道传输到重定向端点时，客户端认证增加了一层安全保护。

* 从受损客户端中恢复，方法是禁用客户端或更改其凭证，从而防止攻击者滥用被盗的刷新令牌。更改一组客户端凭证比撤销整组刷新令牌要快得多。

* 实施认证管理最佳实践，这需要定期轮换凭证。轮换整组刷新令牌可能具有挑战性，而轮换一组客户端凭证则要容易得多。

### 3.2.2. 令牌端点请求

客户端通过使用 form-encoded 序列化格式（per Appendix C.2），在 HTTP 请求内容中使用 UTF-8 字符编码发送以下参数，向令牌端点发出请求：

**"grant_type"**：必需的。客户端用于特定令牌请求的许可类型标识符。本规范定义了 authorization_code、refresh_token 和 client_credentials 值。许可类型决定了令牌请求所需或支持的进一步参数。这些许可类型的详细信息在下面定义。

**"client_id"**：可选的。当使用依赖该参数的某种形式的客户端认证时，或者当 grant_type 需要识别公开客户端时，需要客户端标识符。

机密客户端必须按照第 3.2.1 节中的描述进行认证。

例如，客户端发出以下 HTTPS 请求（仅为显示目的添加额外换行）：

```
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=SplxlOBeZQQYbYS6WxSbIA
&redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb
&code_verifier=3641a2d12d66101249cdf7a79c000c1f8c05d2aafcf14bf146497bed
```

授权服务器必须：

* 要求机密客户端进行客户端认证（或具有其他认证要求的客户端）

* 如果包含客户端认证，则对客户端进行认证

进一步的许可类型特定处理规则适用，并在各许可类型中指定。

### 3.2.3. 令牌端点响应

如果访问令牌请求有效且已授权，授权服务器将颁发访问令牌和可选的刷新令牌。

如果客户端认证失败或无效，授权服务器将返回第 3.2.4 节中描述的错误响应。

授权服务器通过创建 HTTP 响应（根据附录 C.3），使用 [RFC8259] 定义的 application/json 媒体类型，包含以下参数和 HTTP 200 (OK) 状态码来颁发访问令牌和可选的刷新令牌：

**"access_token"**：必需的。授权服务器颁发的访问令牌。

**"token_type"**：必需的。颁发的访问令牌的类型，如第 1.4 节所述。值不区分大小写。

**"expires_in"**：推荐的。一个 JSON 数字，表示访问令牌的生命周期（以秒为单位）。例如，值 3600 表示访问令牌将在响应生成后一小时内到期。如果省略，授权服务器应通过其他方式提供生命周期或记录默认值。请注意，授权服务器可能会提前使访问令牌到期，客户端不得期望访问令牌在提供的生命周期内有效。

**"scope"**：如果与客户端请求的作用域相同，则是推荐的；否则是必需的。访问令牌的作用域，如第 1.4.1 节所述。

**"refresh_token"**：可选的。刷新令牌，可用于根据相应令牌请求中传递的许可获取新的访问令牌。

授权服务器应根据风险评估和自身策略决定是否为某个客户端颁发刷新令牌。如果授权服务器决定不颁发刷新令牌，客户端可以通过重新开始 OAuth 流程来获取新的访问令牌，例如发起新的授权码请求。在这种情况下，授权服务器可以利用 cookie 和持久化许可来优化用户体验。

如果颁发了刷新令牌，这些刷新令牌必须绑定到资源所有者同意的作用域和资源服务器。这是为了防止合法客户端的权限提升并减少刷新令牌泄露的影响。

参数被序列化为 JavaScript 对象表示法（JSON）结构，如附录 C.3 中所述。

授权服务器必须在包含令牌、凭证或其他敏感信息的任何响应中包含 HTTP Cache-Control 响应头字段（见 [RFC9111] 第 5.2 节），值为 no-store。

例如：

```
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-store

{
  "access_token": "2YotnFZFEjr1zCsicMWpAA",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "tGzv3JOkF0XG5Qx2TlKWIA",
  "example_parameter": "example_value"
}
```

客户端必须忽略响应中无法识别的值名称。从授权服务器接收的令牌和其他值的大小是未定义的。客户端应避免对值大小做出假设。授权服务器应记录其颁发的任何值的大小。

### 3.2.4. 令牌端点错误响应

授权服务器使用 HTTP 400（Bad Request）状态码进行响应（除非另有指定），并在响应中包含以下参数：

**"error"**：必需的。来自以下情况的单个 ASCII [USASCII] 错误代码：

* **"invalid_request"**：请求缺少必需的参数、包含不支持的参数值（许可类型除外）、重复参数、包含多个凭证、使用多种机制对客户端进行认证、包含 code_verifier 但授权请求中未发送 code_challenge，或格式不正确。

* **"invalid_client"**：客户端认证失败（例如未知客户端、未包含客户端认证或不支持的认证方法）。授权服务器可以返回 HTTP 401（Unauthorized）状态码以指示支持哪些 HTTP 认证方案。如果客户端尝试通过 Authorization 请求头字段进行认证，授权服务器必须使用 HTTP 401（Unauthorized）状态码进行响应，并包含与客户端使用的认证方案匹配的 WWW-Authenticate 响应头字段。

* **"invalid_grant"**：提供的授权许可（例如授权码、资源所有者凭证）或刷新令牌无效、过期、撤销、不匹配授权请求中使用的重定向 URI，或颁发给了另一个客户端。

* **"unauthorized_client"**：已认证的客户端无权使用此授权许可类型。

* **"unsupported_grant_type"**：授权服务器不支持该授权许可类型。

* **"invalid_scope"**：请求的作用域无效、未知、格式错误或超过资源所有者授予的作用域。

错误参数的值不得包含 %x20-21 / %x23-5B / %x5D-7E 集合之外的字符。

**"error_description"**：可选的。提供附加信息的可读性 ASCII [USASCII] 文本，用于帮助客户端开发者理解发生的错误。error_description 参数的值不得包含 %x20-21 / %x23-5B / %x5D-7E 集合之外的字符。

**"error_uri"**：可选的。一个 URI，标识包含错误信息的可读性网页，用于向客户端开发者提供有关错误的附加信息。error_uri 参数的值必须符合 URI-reference 语法，因此不得包含 %x21 / %x23-5B / %x5D-7E 集合之外的字符。

参数使用附录 C.3 中定义的 application/json 媒体类型包含在 HTTP 响应的内容中。

例如：

```
HTTP/1.1 400 Bad Request
Content-Type: application/json
Cache-Control: no-store

{
 "error": "invalid_request"
}
```
