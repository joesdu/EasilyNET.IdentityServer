# 8. 原生应用程序

原生应用程序是安装在资源所有者使用的设备上并执行的客户端（即桌面应用程序或原生移动应用程序）。原生应用程序需要与安全、平台能力和整体最终用户体验相关的特殊考虑。

本节中的指导主要在原生移动应用程序而非桌面应用程序的背景下。原生移动平台在为本开发者提供的与此处描述的 OAuth 流程相关的能力方面比桌面平台更成熟。虽然指导主要针对移动应用程序，但其大部分通常也适用于桌面应用程序。

授权端点需要客户端与资源所有者的用户代理之间的交互。最佳当前实践是在外部用户代理（通常是浏览器）中执行 OAuth 授权请求，而不是嵌入式用户代理（例如使用 web-views 实现的用户代理）。

原生应用程序可以通过几种不同方式捕获授权服务器的响应，每种方式具有不同的安全属性。例如，使用带有"应用声明的 URL"的重定向 URI 或在操作系统注册的自定义 URL 方案来将客户端作为处理程序调用、手动复制粘贴凭证、运行本地 Web 服务器、安装用户代理扩展，或通过提供标识客户端控制下的服务器托管资源的重定向 URI 来使响应对原生应用程序可用。

以前，原生应用程序通常使用嵌入式用户代理（通常使用 web-views 实现）进行 OAuth 授权请求。这种方法有许多缺点，包括主机应用程序能够复制用户凭证和 Cookie，以及用户需要在每个应用程序中从头开始认证。参见第 8.5.1 节，深入了解使用嵌入式用户代理进行 OAuth 的缺点。

使用系统浏览器进行授权请求的原生应用程序授权更安全，可以利用设备上用户的认证状态。能够使用浏览器中的现有认证会话可以实现单点登录，因为用户不需要每次使用新应用程序时都向授权服务器进行认证（除非授权服务器策略要求）。

支持原生应用程序与浏览器之间的授权流程可以在不更改 OAuth 协议本身的情况下实现，因为 OAuth 授权请求和响应已经根据 URI 定义。这包含了可用于应用间通信的 URI。某些假设所有客户端都是机密 Web 客户端的 OAuth 服务器实现需要添加对公共原生应用客户端及其使用的重定向 URI 类型的理解，以支持此最佳实践。

## 8.1. 原生应用程序的客户端认证

作为应用的一部分静态包含的密钥分发给多个用户，这些密钥不是机密密钥，因为一个用户可能检查其副本并了解共享密钥。因此，授权服务器不得要求原生应用客户端使用共享密钥进行客户端认证，因为这除了已由 client_id 请求参数提供的客户端标识之外没有任何价值。

仍然要求原生应用客户端使用静态包含共享密钥的授权服务器必须将客户端视为公共客户端（如第 2.1 节所定义），并且不接受该密钥作为客户端身份的证明。如果没有额外措施，此类客户端容易受到客户端冒充（见第 7.3.1 节）。

### 8.1.1. 原生应用客户端的注册

除非使用动态客户端注册 [RFC7591] 等机制来配置每个实例的凭证，否则原生应用程序被归类为公共客户端（如第 2.1 节所定义），并且必须作为此类向授权服务器注册。授权服务器必须在客户端注册详细信息中记录客户端类型，以便相应地识别和处理请求。

### 8.1.2. 原生应用认证

草案规范 [I-D.ietf-oauth-attestation-based-client-auth] 定义了一种机制，原生应用程序可使用该机制获取密钥绑定认证以向授权服务器或资源服务器进行认证。这可以提供更高级别的移动应用程序身份保证。

## 8.2. 在原生应用程序中使用应用间 URI 通信进行 OAuth

正如 URIs 用于 Web 上的 OAuth 以启动授权请求并将授权响应返回请求网站一样，URI 也可用于原生应用程序在设备浏览器中启动授权请求并将响应返回请求的原生应用程序。

通过采用与 Web 上用于 OAuth 的相同方法，在 Web 上下文中看到的单点登录会话可用性和单独认证上下文安全性的好处同样可以在原生应用程序上下文中获得。重用相同方法也降低了实现复杂性，并通过依赖不特定于任何特定平台的标准 Web 流程来提高互操作性。

原生应用程序必须使用外部用户代理执行 OAuth 授权请求。这是通过在浏览器中打开授权请求（第 8.3 节中详细说明）并使用将授权响应返回原生应用程序的重定向 URI（第 8.4 节中定义）来实现的。

## 8.3. 从原生应用程序发起授权请求

需要用户授权的原生应用程序使用第 4.1 节中的授权码许可类型创建授权请求 URI，使用能够被原生应用程序接收的重定向 URI。

原生应用程序授权请求的重定向 URI 功能类似于基于 Web 的授权请求。原生应用程序使用的重定向 URI 不是将响应返回到 OAuth 客户端的服务器，而是将响应返回到应用程序。第 8.4 节记录了在不同平台上将授权响应返回原生应用程序的几种重定向 URI 选项。任何允许应用程序接收 URI 并检查其参数的重定向 URI 都是可行的。

在构建授权请求 URI 后，应用程序使用特定于平台的 API 在外部用户代理中打开 URI。通常，使用的外部用户代理是默认浏览器，即配置为在系统上处理 http 和 https 方案 URI 的应用程序；但是，可以使用不同的浏览器选择标准和其他类别的外部用户代理。

此最佳实践将浏览器作为原生应用程序推荐的外部用户代理。专门为用户授权设计并能够像浏览器一样处理授权请求和响应的外部用户代理也可以使用。其他外部用户代理，例如由授权服务器提供的原生应用程序，可能满足此最佳实践中规定的标准，包括使用相同的重定向 URI 属性，但其使用超出本规范范围。

某些平台支持称为"应用内浏览器标签"的功能，应用程序可以在应用上下文中呈现浏览器标签而不切换应用程序，但仍保留浏览器的关键优势，如共享认证状态和安全上下文。在支持这些功能的平台上，出于可用性原因，建议应用程序使用应用内浏览器标签进行授权请求。

## 8.4. 在原生应用程序中接收授权响应

原生应用程序有几种重定向 URI 选项可用于从浏览器接收授权响应，这些选项的可用性和用户体验因平台而异。

### 8.4.1. 声称的"https"方案 URI 重定向

某些操作系统（特别是移动操作系统）允许应用程序声明其控制的域名中的 https URI（参见 [RFC9110] 第 4.2.2 节）。当浏览器遇到声称的 URI 时，不是加载页面在浏览器中，而是使用作为启动参数提供的 URI 启动原生应用程序。

此类 URI 可被原生应用程序用作重定向 URI。对于授权服务器来说，它们与常规基于 Web 的客户端重定向 URI 无法区分。例如：

```
https://app.example.com/oauth2redirect/example-provider
```

由于重定向 URI 本身不足以区分公共原生应用客户端和机密 Web 客户端，第 8.1.1 节要求在客户端注册期间记录客户端类型，以使服务器能够确定客户端类型并相应处理。

与应用声明的 https 方案重定向 URI 相比其他原生应用重定向选项有一些优势，因为操作系统向授权服务器保证了目标应用程序的身份。因此，原生应用程序应尽可能使用它们而不是其他选项。

### 8.4.2. 回环接口重定向

能够在回环网络接口上打开端口而不需要特殊权限的原生应用程序（通常在桌面操作系统上）可以使用回环接口接收 OAuth 重定向。

回环重定向 URI 使用 http 方案，并使用回环 IP 字面量和客户端正在监听的任何端口构建。

即，对于 IPv4 为 `http://127.0.0.1:{port}/{path}`，对于 IPv6 为 `http://[::1]:{port}/{path}`。以下是使用随机分配端口的 IPv4 回环接口的重定向示例：

```
http://127.0.0.1:51004/oauth2redirect/example-provider
```

以下是使用随机分配端口的 IPv6 回环接口的重定向示例：

```
http://[::1]:61023/oauth2redirect/example-provider
```

虽然使用 localhost 名称（即 `http://localhost:{port}/{path}`）的重定向 URI 的功能类似于回环 IP 重定向，但不建议使用 localhost。指定带有回环 IP 字面量而不是 localhost 的重定向 URI 可避免意外监听回环接口以外的网络接口。它也不太容易受到用户设备上客户端防火墙和错误配置的主机名解析的影响。

授权服务器必须允许为回环 IP 重定向 URI 在请求时指定任何端口，以适应客户端在请求时从操作系统获取的可用临时端口。

客户端不应假设设备支持特定版本的互联网协议。建议客户端尝试绑定到 IPv4 和 IPv6 回环接口，并使用可用的那个。

### 8.4.3. 私有使用 URI 方案重定向

许多移动和桌面计算平台通过允许应用程序注册私有使用 URI 方案（有时俗称为"自定义 URL 方案"）（如 com.example.app）来支持通过 URI 进行应用间通信。当浏览器或其他应用程序尝试加载带有私有使用 URI 方案的 URI 时，注册它的应用程序会启动以处理请求。

许多支持私有使用 URI 方案的环境不提供声明方案和阻止其他方使用另一个应用程序方案的机制。因此，使用私有使用 URI 方案的客户端容易受到对其重定向 URI 的潜在攻击，因此此选项应仅在前面提到的更安全的选项不可用时使用。

要使用私有使用 URI 方案重定向执行授权请求，原生应用程序使用标准授权请求启动浏览器，但其中一个重定向 URI 使用了它向操作系统注册的私有使用 URI 方案。

在为应用程序选择 URI 方案时，应用程序必须使用基于其控制的域名的 URI 方案，以相反顺序表示，如 [RFC7595] 第 3.8 节为私有使用 URI 方案所推荐。

例如，控制域名 app.example.com 的应用程序可以使用 com.example.app 作为其方案。某些授权服务器根据域名分配客户端标识符，例如 client1234.usercontent.example.net，当以相同方式反转时，也可以用作该方案的域名。

由于没有命名机构用于私有使用 URI 方案重定向，根据 [RFC3986] 第 3.2 节，在方案组件后只有一个斜杠（/）。使用私有使用 URI 方案的重定向 URI 的完整示例：

```
com.example.app:/oauth2redirect/example-provider
```

当授权服务器完成请求时，它会像往常一样重定向到客户端的重定向 URI。由于重定向 URI 使用私有使用 URI 方案，它会导致操作系统启动原生应用程序，将 URI 作为启动参数传入。然后，原生应用程序使用正常流程处理授权响应。

## 8.5. 原生应用程序中的安全考虑

### 8.5.1. 原生应用程序中的嵌入式用户代理

嵌入式用户代理是授权原生应用程序的技术上可行的方法。默认情况下，这些嵌入式用户代理对授权服务器的第三方不安全，因为托管嵌入式用户代理的应用程序可以访问用户的完整认证凭证，而不仅仅是应用程序所需的 OAuth 授权许可。它们通常也被操作系统沙箱化，并且依赖于 Web 来源的机制（如 WebAuthn）被禁用。

在典型的基于 web-view 的嵌入式用户代理实现中，主机应用程序可以记录登录表单中输入的每个击键以捕获用户名和密码，自动提交表单以绕过用户同意，并复制会话 Cookie 并使用它们作为用户执行认证操作。

即使由与授权服务器同一方的可信应用程序使用，嵌入式用户代理也通过访问比所需更强大的凭证而违反最小权限原则，可能会增加攻击面。

鼓励用户在嵌入式用户代理中输入凭证，而没有浏览器通常具有的通常地址栏和可见证书验证功能，使用户无法知道他们是否在合法站点登录；即使合法，这也会训练他们在不首先验证站点的情况下输入凭证是可以的。

除了安全问题，嵌入式用户代理不与其他应用程序或浏览器共享认证状态，需要用户为每个授权请求登录，这通常被认为是一种较差用户体验。

### 8.5.2. 原生应用程序中的虚假外部用户代理

发起授权请求的原生应用程序对用户界面有很大程度的控制，可能能够呈现虚假的外部用户代理，即看起来像外部用户代理的嵌入式用户代理。

当所有好的行为者都使用外部用户代理时，优势是安全专家可以检测坏行为者，因为任何伪造外部用户代理的人都是可证明的坏人。另一方面，如果好的和坏的行为者都使用嵌入式用户代理，坏的行为者不需要伪造任何东西，使它们更难被检测。一旦检测到恶意应用程序，可能可以使用此知识在恶意软件扫描软件中黑名单化应用程序的签名，采取移除操作（在应用程序商店分发应用程序的情况下）和其他步骤以减少恶意应用程序的影响和传播。

授权服务器也可以通过要求仅适用于真正的外部用户代理的认证因素来直接保护免受虚假外部用户代理的侵害。

特别关注安全性的用户在应用内浏览器标签中进行操作时，也可以采取额外步骤从应用内浏览器标签在完整浏览器中打开请求并在那里完成授权，因为大多数应用内浏览器标签模式的实现都提供此功能。

### 8.5.3. 原生应用程序中的恶意外部用户代理

如果恶意应用程序能够将自己配置为操作系统中 https 方案 URI 的默认处理程序，它将能够拦截使用默认浏览器的授权请求，并滥用这种信任地位进行恶意目的（如对用户进行钓鱼）。

此攻击不限于 OAuth；以此方式配置的恶意应用程序将对用户构成超出原生应用程序 OAuth 使用范围的一般性和持续性风险。许多操作系统通过要求用户明确操作来更改 http 和 https 方案 URI 的默认处理程序来缓解此问题。

### 8.5.4. 原生应用程序中的回环重定向注意事项

回环接口重定向 URI 可以使用 http 方案（即不使用 TLS）。这对于回环接口重定向 URI 是可接受的，因为 HTTP 请求永远不会离开设备。

客户端仅应在启动授权请求时打开网络端口，并在返回响应后立即关闭它。

客户端应仅监听回环网络接口，以避免其他网络参与者的干扰。

客户端应使用回环 IP 字面量而不是字符串 localhost，如第 8.4.2 节所述。
