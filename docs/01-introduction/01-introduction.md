# 1. 介绍

OAuth 在客户端-服务器认证模型中引入了一个授权层，将客户端的角色与资源所有者的角色分离。在 OAuth 中，客户端请求访问由资源所有者控制并由资源服务器托管的资源。客户端不是使用资源所有者的凭证来访问受保护资源，而是获取一个访问令牌——一种代表特定访问属性（如作用域和生命周期）的凭证。访问令牌由授权服务器在获得资源所有者批准后颁发给客户端。客户端使用访问令牌来访问由资源服务器托管的受保护资源。

在旧的、限制更多的客户端-服务器认证模型中，客户端通过使用资源所有者的凭证向服务器进行认证来请求服务器上的访问限制资源（受保护资源）。为了让应用程序访问受限制资源，资源所有者与应用程序共享其凭证。这会产生以下几个问题和限制：

* 应用程序需要存储资源所有者的凭证以供将来使用，通常是明文密码。

* 尽管密码固有的安全弱点，服务器仍需支持密码认证。

* 应用程序对资源所有者的受保护资源获得了过于广泛的访问权限，使资源所有者无法限制访问期限或限制访问特定子集的资源。

* 尽管遵循了最佳安全实践，资源所有者仍经常将密码与其他不相关的服务重用。这种密码重用意味着一个服务中的漏洞或暴露可能对完全不相关的服务产生安全影响。

* 资源所有者无法撤销对单个应用程序的访问权限而不撤销对所有第三方的访问权限，必须通过更改密码来实现。

* 任何应用程序的泄露都会导致最终用户密码及该密码保护的所有数据的泄露。

OAuth 的一个使用示例是：最终用户（资源所有者）授予一个财务管理服务（客户端）访问其存储在银行服务（资源服务器）上的敏感交易记录的权限，而不与财务管理服务共享其用户名和密码。相反，他们直接在其金融机构服务器（授权服务器）上进行认证，授权服务器向财务管理服务颁发特定委托凭证（访问令牌）。

这种关注点分离还提供了使用更先进的用户认证方法的能力，例如多因素认证甚至无密码认证，而无需对应用程序进行任何修改。由于所有用户认证逻辑都由授权服务器处理，应用程序不需要关心实现任何特定认证机制的细节。这使得授权服务器能够管理用户认证策略，甚至可以在将来更改它们，而无需与应用程序协调更改。

授权层还可以简化资源服务器确定请求是否已授权的方式。传统上，在对客户端进行认证后，每个资源服务器都会评估策略，以计算客户端在每次 API 调用时是否获得授权。在分布式系统中，策略需要同步到所有资源服务器，或者资源服务器必须调用中央策略服务器来处理每个请求。在 OAuth 中，策略的评估仅在授权服务器创建新的访问令牌时执行。如果授权访问在访问令牌中表示，则资源服务器不再需要评估策略，只需验证访问令牌。当应用程序代表资源所有者行事或代表自身行事时，这种简化适用。

OAuth 是一个授权协议，而不是认证协议，因为 OAuth 没有定义实现用户认证所需的必要组件。如果目标是认证用户，则需要认证协议。一个例子是 OpenID Connect [OpenID.Connect]，它建立在 OAuth 之上，提供了认证协议所需的安全特性和必要组件。

访问令牌代表授予客户端的授权。客户端向专有 API 提交访问令牌以获取资源所有者的用户标识符，然后使用 API 结果作为用户认证的代理，这是一种常见做法。这种做法不属于 OAuth 标准或安全考虑的范围，资源所有者可能并未考虑过。在采用此做法之前，实现者应仔细查阅资源服务器的文档。

本规范设计用于与 HTTP [RFC9110] 一起使用。将 OAuth 用于除 HTTP 以外的任何协议不在本规范范围内。

自 2012 年 10 月发布 OAuth 2.0 授权框架 [RFC6749] 以来，它已通过 OAuth 2.0 原生应用 [RFC8252]、OAuth 安全最佳当前实践 [RFC9700] 和 OAuth 2.0 浏览器应用 [I-D.ietf-oauth-browser-based-apps] 进行了更新。OAuth 2.0 授权框架：持有者令牌使用 [RFC6750] 也已通过 [RFC9700] 进行了更新。本标准跟踪规范整合了所有这些文档中的信息，并删除了 [RFC9700] 中发现的不安全的功能。

## 1.1. 角色

OAuth 定义了四个角色：

**"资源所有者"（resource owner）**：有权授予对受保护资源的访问权限的实体。当资源所有者是人时，称为最终用户。有时缩写为"RO"。

**"资源服务器"（resource server）**：托管受保护资源的服务器，能够使用访问令牌接受和响应受保护资源请求。资源服务器通常可通过 API 访问。有时缩写为"RS"。

**"客户端"（client）**：代表资源所有者并经其授权发起受保护资源请求的应用程序。"客户端"一词不暗示任何特定的实现特性（例如，应用程序是在服务器、台式机还是其他设备上执行）。

**"授权服务器"（authorization server）**：在成功认证资源所有者并获得授权后，向客户端颁发访问令牌的服务器。有时缩写为"AS"。

本规范的大部分内容定义了客户端与授权服务器之间以及客户端与资源服务器之间的交互。

授权服务器与资源服务器之间的交互不在本规范范围内，但已定义了几个扩展以提供资源服务器与授权服务器之间的互操作性选项。授权服务器可以是与资源服务器相同的服务器，也可以是单独的实体。一个授权服务器可以颁发被多个资源服务器接受的访问令牌。

资源所有者与授权服务器之间的交互（例如，最终用户如何在授权服务器上进行身份验证）也不在本规范范围内，但有一些例外，例如围绕提示最终用户同意的安全考虑。

当资源所有者是最终用户时，用户将与客户端交互。当客户端是基于 Web 的应用程序时，用户将通过用户代理与客户端交互（如 [RFC9110] 第 3.5 节中所述）。当客户端是原生应用程序时，用户将通过操作系统直接与客户端交互。详见第 2.1 节。

## 1.2. 协议流程

```

|        |--(1)- Authorization Request ->|   Resource    |
|        |                               |     Owner     |
|        |<-(2)-- Authorization Grant ---|               |
|        |                               +---------------+
|        |
|        |                               +---------------+
|        |--(3)-- Authorization Grant -->| Authorization |
| Client |                               |     Server    |
|        |<-(4)----- Access Token -------|               |
|        |                               +---------------+
|        |
|        |                               +---------------+
|        |--(5)----- Access Token ------>|    Resource   |
|        |                               |     Server    |
|        |<-(6)--- Protected Resource ---|               |
+--------+                               +---------------+

               Figure 1: Abstract Protocol Flow
```

图 1 中说明的抽象 OAuth 2.1 流程描述了四个角色之间的交互，包括以下步骤：

1. 客户端向资源所有者请求授权。授权请求可以直接向资源所有者提出（如所示），或者最好通过授权服务器作为中介间接提出。

2. 客户端收到一个授权许可，这是一种代表资源所有者授权的凭证，使用本规范定义的授权许可类型之一或扩展许可类型来表示。授权许可类型取决于客户端用于请求授权的方法以及授权服务器支持的类型。

3. 客户端通过向授权服务器进行认证并出示授权许可来请求访问令牌。

4. 授权服务器认证客户端并验证授权许可，如果有效，则颁发访问令牌。

5. 客户端通过出示访问令牌向资源服务器请求受保护资源。

6. 资源服务器验证访问令牌，如果有效，则为请求提供服务。

客户端从资源所有者获取授权许可的首选方法（如图 1 和图 3 中所示的步骤（1）和（2））是使用授权服务器作为中介。

## 1.3. 授权许可

授权许可代表资源所有者的授权（访问其受保护资源），客户端使用它来获取访问令牌。本规范定义了三种许可类型——授权码、刷新令牌和客户端凭证——以及用于定义其他类型的扩展机制。

### 1.3.1. 授权码

授权码是用于获取访问令牌的临时凭证。客户端不是直接从资源所有者请求授权，而是将资源所有者引导至授权服务器（通过其用户代理），授权服务器又引导资源所有者带着授权码返回客户端。然后客户端可以将授权码兑换为访问令牌。

在带着授权码引导资源所有者返回客户端之前，授权服务器会对资源所有者进行认证，并可能请求资源所有者的同意或以其他方式告知他们客户端的请求。因为资源所有者仅在授权服务器进行认证，所以资源所有者的凭证永远不会与客户端共享，客户端也不需要了解任何额外的认证步骤，例如多因素认证或委托账户。

授权码提供了几个重要的安全优势，例如能够对客户端进行认证，以及能够将访问令牌直接传输给客户端，而无需通过资源所有者的用户代理，从而可能将其暴露给其他人，包括资源所有者。

### 1.3.2. 刷新令牌

刷新令牌是用于获取访问令牌的凭证。刷新令牌可以由授权服务器颁发给客户端，用于在当前访问令牌变得无效或过期时获取新的访问令牌，或获取具有相同或更窄作用域的额外访问令牌（访问令牌可能比资源所有者授权的具有更短的生命周期和更少的特权）。是否颁发刷新令牌由授权服务器自行决定，可以基于客户端的属性、请求的属性、授权服务器内部的策略或任何其他标准。如果授权服务器颁发刷新令牌，则在颁发访问令牌时将其包含在内（即图 2 中的步骤（2））。刷新令牌的生命周期也由授权服务器自行决定。

刷新令牌是一个字符串，表示授予客户端的授权。该字符串对客户端而言是不透明的。刷新令牌可以是一个用于检索授权信息的标识符，或者可以将此信息编码到字符串本身中。与访问令牌不同，刷新令牌仅用于授权服务器，永远不会发送到资源服务器。

```
+--------+                                           +---------------+
|        |--(1)------- Authorization Grant --------->|               |
|        |                                           |               |
|        |<-(2)----------- Access Token -------------|               |
|        |               & Refresh Token             |               |
|        |                                           |               |
|        |                            +----------+   |               |
|        |--(3)---- Access Token ---->|          |   |               |
|        |                            |          |   |               |
|        |<-(4)- Protected Resource --| Resource |   | Authorization |
| Client |                            |  Server  |   |     Server    |
|        |--(5)---- Access Token ---->|          |   |               |
|        |                            |          |   |               |
|        |<-(6)- Invalid Token Error -|          |   |               |
|        |                            +----------+   |               |
|        |                                           |               |
|        |--(7)----------- Refresh Token ----------->|               |
|        |                                           |               |
|        |<-(8)----------- Access Token -------------|               |
+--------+           & Optional Refresh Token        +---------------+

          Figure 2: Refreshing an Expired Access Token
```

图 2 中说明的流程包括以下步骤：

1. 客户端通过向授权服务器进行认证并出示授权许可来请求访问令牌。

2. 授权服务器认证客户端并验证授权许可，如果有效，则颁发访问令牌和可选的刷新令牌。

3. 客户端通过出示访问令牌向资源服务器发起受保护资源请求。

4. 资源服务器验证访问令牌，如果有效，则为请求提供服务。

5. 步骤（3）和（4）重复，直到访问令牌过期。如果客户端知道访问令牌已过期，则跳到步骤（7）；否则，它会发起另一个受保护资源请求。

6. 由于访问令牌无效，资源服务器返回无效令牌错误。

7. 客户端通过出示刷新令牌并提供客户端认证（如果已颁发凭证）来请求新的访问令牌。客户端认证要求基于客户端类型和授权服务器策略。

8. 授权服务器认证客户端并验证刷新令牌，如果有效，则颁发新的访问令牌（以及可选的新刷新令牌）。

请注意，不需要将刷新令牌的生命周期传达给客户端，因为客户端无法根据此知识做不同的事情。此外，授权服务器可能选择使用动态生命周期（例如，只要刷新令牌至少每 7 天使用一次，刷新令牌到期时间就会延长），或者授权服务器可能会在任何原因（例如用户撤销了应用程序的访问权限）的情况下在刷新令牌预定到期日期之前撤销它。这意味着客户端已经需要处理刷新令牌在任意时间过期的情况。

无论刷新令牌过期的原因或时间如何，客户端获取新令牌的途径只有一条，即从头开始新的 OAuth 流程。因此，没有定义属性来将刷新令牌的过期情况传达给客户端。

### 1.3.3. 客户端凭证

当授权范围限于客户端控制的受保护资源，或限于先前与授权服务器约定的受保护资源时，客户端凭证或其他形式的客户端认证（例如，用于签名 JWT 的私钥，如 [RFC7523] 及其更新 [I-D.ietf-oauth-rfc7523bis] 中所述）可用作授权许可。客户端凭证用于客户端基于与授权服务器预先安排的授权来请求访问受保护资源。

## 1.4. 访问令牌

访问令牌是用于访问受保护资源的凭证。访问令牌是颁发给客户端的授权的字符串表示形式。

该字符串对客户端而言是不透明的，即使它具有结构。客户端不得期望能够解析访问令牌值。授权服务器不需要使用一致的访问令牌编码或格式，只要资源服务器能够理解即可。

资源所有者授予客户端的访问权限由授权服务器创建的访问令牌表示。访问令牌的有效期较短，以减少泄露访问令牌的影响范围。访问令牌的有效期由授权服务器设置。

根据授权服务器的实现，令牌字符串可用于资源服务器检索授权信息，或者令牌可以以可验证的方式自包含授权信息（即，由签名数据载荷组成的令牌字符串）。令牌检索机制的一个例子是令牌 introspection [RFC7662]，其中 RS 在客户端提交的令牌上调用 AS 上的端点来验证令牌。结构化令牌格式的一个例子是 JWT Profile for Access Tokens [RFC9068]，一种将访问令牌数据编码和签名为 JSON Web Token [RFC7519] 的方法。

可能需要额外的认证凭证（超出本规范范围）才能让客户端使用访问令牌。这通常被称为发送者约束的访问令牌，例如 DPoP [RFC9449] 和 Mutual TLS Certificate-Bound Access Tokens [RFC8705]。

访问令牌提供了一个抽象层，用资源服务器理解的单一令牌替换不同的授权构造（例如用户名和密码）。这种抽象能够颁发比用于获取它们的授权许可更具限制性的访问令牌，也消除了资源服务器需要理解各种认证方法的必要性。

访问令牌可以具有不同的格式、结构和利用方法（例如加密属性），具体取决于资源服务器的安全要求。访问令牌属性和用于访问受保护资源的方法可以扩展到本规范描述的范围之外。

访问令牌（以及任何机密访问令牌属性）在传输和存储过程中必须保持机密，仅在授权服务器、访问令牌有效的资源服务器以及颁发给的客户端之间共享。

授权服务器必须确保访问令牌不能被未经授权的一方生成、修改或猜测以产生有效访问令牌。

### 1.4.1. 访问令牌作用域

访问令牌旨在颁发给具有比授予访问权限的用户更少特权的客户端。这被称为有限"作用域"访问令牌。授权服务器和资源服务器可以使用此作用域机制来限制特定客户端可以拥有的资源类型或访问级别。

例如，客户端可能只需要对用户资源的"只读"访问权限，但不需要更新资源，因此客户端可以请求授权服务器定义的只读作用域，并获取一个不能用于更新资源的访问令牌。这需要授权服务器、资源服务器和客户端之间的协调。授权服务器为客户端提供请求特定作用域的能力，并将这些作用域关联到颁发给客户端的访问令牌。然后，资源服务器在收到有限作用域访问令牌时负责强制执行作用域。

OAuth 没有定义任何作用域值，作用域由授权服务器或 OAuth 的扩展或概要文件定义。定义作用域的一个扩展是 [OpenID.Connect]，它定义了一组提供对用户配置文件信息细粒度访问的作用域。建议避免定义与已知扩展的作用域冲突的自定义作用域。

要请求有限作用域访问令牌，客户端根据所使用的许可类型，在授权端点或令牌端点使用 scope 请求参数。反过来，授权服务器使用 scope 响应参数通知客户端颁发的访问令牌的作用域。

scope 参数的值表示为大小写敏感的字符串的空格分隔列表。这些字符串由授权服务器定义。如果值包含多个空格分隔的字符串，则它们的顺序无关紧要，每个字符串都会为请求的作用域添加额外的访问范围。

```
scope       = scope-token *( SP scope-token )
scope-token = 1*( %x21 / %x23-5B / %x5D-7E )
```

授权服务器可以根据授权服务器策略或资源所有者的指示完全或部分忽略客户端请求的作用域。如果颁发的访问令牌作用域与客户端请求的作用域不同，授权服务器必须在令牌响应（第 3.2.3 节）中包含 scope 响应参数，以通知客户端实际授予的作用域。

如果客户端在请求授权时省略了 scope 参数，授权服务器必须使用预定义的默认值处理请求，或拒绝请求并指示无效作用域。授权服务器应记录其作用域要求和默认值（如果已定义）。

### 1.4.2. 持有者令牌

持有者令牌是一种安全令牌，具有以下特性：拥有该令牌的任何一方（"持有者"）都可以以与任何其他持有者相同的方式使用该令牌。使用持有者令牌不需要持有者证明拥有加密密钥材料（占有证明）。

持有者令牌可以使用 DPoP [RFC9449] 和 mTLS [RFC8705] 等占有证明规范进行增强，以提供占有证明特性。

为了防止访问令牌泄露，客户端与资源服务器之间的通信交互必须使用第 1.5 节中描述的保密性和完整性保护。

对持有者令牌的具体结构或格式没有要求。如果持有者令牌是对授权信息的引用，则此类引用对于攻击者来说必须不可猜测，例如使用足够长的加密随机字符串。如果持有者令牌使用编码机制将授权信息包含在令牌本身中，则访问令牌必须使用足够的完整性保护来防止令牌被修改。访问令牌的编码和签名机制的一个示例在 JSON Web Token Profile for Access Tokens [RFC9068] 中描述。

### 1.4.3. 发送者约束访问令牌

发送者约束访问令牌将访问令牌的使用绑定到特定的发送者。该发送者有义务在接收方（例如资源服务器）接受该访问令牌之前，证明知道某个密钥。

授权服务器和资源服务器应使用发送者约束访问令牌的机制，例如 OAuth 占有证明（DPoP）[RFC9449] 或 OAuth 2.0 Mutual TLS [RFC8705]。请参阅 [RFC9700] 第 4.10.1 节，以防止滥用被盗和泄露的访问令牌。

建议在客户端和资源服务器之间使用端到端 TLS。如果 TLS 流量需要在中间件终止，请参阅 [RFC9700] 第 4.13 节以获取进一步的安全建议。

## 1.5. 通信安全

实现必须使用提供通信认证、完整性和保密性的机制，例如传输层安全 [RFC8446]，以保护明文凭证和令牌（在内容中或头字段中）免受窃听，从而防止重放（例如，请参阅第 2.4.1 节、第 7.5.1 节、第 3.2 节和第 1.4.2 节）。

所有 OAuth 协议 URL（AS、RS 和 Client 公开的 URL）必须使用 https 方案，但回环接口重定向 URI 除外，它们可以使用 http 方案。使用 https 时，必须根据 [RFC9110] 第 4.3.4 节检查 TLS 证书。在编写本文时，TLS 版本 1.3 [RFC8446] 是最新版本。

实现也可以支持满足其安全要求的其他传输层安全机制。

TLS 版本和算法的标识不在本规范范围内。请参阅 [BCP195] 获取传输层安全的最新建议，以及相关规范以获取证书验证和其他安全考虑。

## 1.6. HTTP 重定向

本规范广泛使用 HTTP 重定向，其中客户端或授权服务器将资源所有者的用户代理引导到另一个目的地。虽然本规范中的示例显示了 HTTP 302 状态码的使用，但允许使用用户代理可用的任何其他方法来完成此重定向，但 HTTP 307 除外，这被视为实现细节。详见第 7.5.3 节。

## 1.7. 互操作性

OAuth 2.1 提供了一个具有明确定义的安全属性的丰富授权框架。

本规范将一些必需的组件部分或完全未定义（例如客户端注册、授权服务器能力、端点发现）。某些这些行为在可选扩展中定义，实现可以选择使用，例如：

* [RFC8414]：授权服务器元数据，定义客户端可用于查找与特定 OAuth 服务器交互所需信息的端点

* [RFC7591]：动态客户端注册，提供一种向授权服务器以编程方式注册客户端的机制

* [RFC7592]：动态客户端管理，提供一种更新动态注册的客户端信息的机制

* [RFC7662]：令牌 introspection，定义资源服务器获取访问令牌信息的机制

请参阅附录 D 以获取本规范发布时已知的当前扩展列表。

## 1.8. 与 OAuth 2.0 的兼容性

OAuth 2.1 与 OAuth 2.0 兼容，同时应用了已知最佳当前实践的扩展和限制。具体来说，OAuth 2.0 核心中未指定的功能（如 PKCE）在 OAuth 2.1 中是必需的。此外，OAuth 2.0 中可用的一些功能（如隐式或资源所有者凭证许可类型）未在 OAuth 2.1 中指定。此外，OAuth 2.0 中允许的一些行为在 OAuth 2.1 中受到限制，例如 OAuth 2.1 要求的重定向 URI 的严格字符串匹配。

有关与 OAuth 2.0 的差异的更多详细信息，请参见第 10 节。

## 1.9. 符号约定

当这些词以全大写形式出现时，本文档中的关键词"MUST"、"MUST NOT"、"REQUIRED"、"SHALL"、"SHALL NOT"、"SHOULD"、"SHOULD NOT"、"RECOMMENDED"、"NOT RECOMMENDED"、"MAY"和"OPTIONAL"应按照 BCP 14 [RFC2119] [RFC8174] 中的描述进行解释。

本规范使用 [RFC5234] 的增强巴科斯-诺尔形式（ABNF）表示法。此外，URI-reference 规则来自"统一资源标识符（URI）：通用语法" [RFC3986]。

某些安全相关术语应按照 [RFC4949] 中定义的含义来理解。这些术语包括但不限于"攻击"、"认证"、"授权"、"证书"、"保密性"、"凭证"、"加密"、"身份"、"签名"、"信任"、"验证"和"验证"。

"content"一词应按照 [RFC9110] 第 6.4 节中的描述进行解释。

"user agent"一词应按照 [RFC9110] 第 3.5 节中的描述进行解释。

除非另有说明，否则所有协议参数名称和值都区分大小写。
