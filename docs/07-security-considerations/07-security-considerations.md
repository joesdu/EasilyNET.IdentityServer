# 7. 安全考虑

作为一个灵活且可扩展的框架，OAuth 的安全考虑取决于许多因素。以下章节为实现者提供了安全指南，重点关注第 2.1 节中描述的三种客户端配置文件：Web 应用程序、基于浏览器的应用程序和原生应用程序。

[RFC6819] 和 [RFC9700] 提供了全面的 OAuth 安全模型和分析，以及协议设计背景。

## 7.1. 访问令牌安全考虑

### 7.1.1. 安全威胁

以下列表列出了针对使用某种形式令牌协议的几种常见威胁。此威胁列表基于 NIST Special Publication 800-63 [NIST800-63]。

### 7.1.1.1. 访问令牌制造/修改

攻击者可能生成伪造的访问令牌或修改现有令牌的内容（如认证或属性语句），导致资源服务器向客户端授予不当访问权限。例如，攻击者可能修改令牌以延长有效期；恶意客户端可能修改断言以获取不应查看的信息。

### 7.1.1.2. 访问令牌信息泄露

访问令牌可能包含敏感信息的认证和属性语句。

如果应阻止客户端观察访问令牌的内容，则必须应用内容加密。

由于 Cookie 默认以明文传输，其中包含的任何信息都有泄露风险：持有者令牌不得存储在可以明文发送的 Cookie 中。有关 Cookie 的安全考虑，请参见 [RFC6265] 第 7 和 8 节。

### 7.1.1.3. 访问令牌重定向

攻击者使用为某个资源服务器生成的访问令牌来访问误认为该令牌面向的不同资源服务器。

### 7.1.1.4. 访问令牌重放

攻击者尝试使用已在该资源服务器上使用过的访问令牌。

### 7.1.2. 威胁缓解

通过使用数字签名保护访问令牌内容，并遵循签名密钥管理最佳实践（如定期密钥轮换），可以减轻大范围威胁。

或者，持有者令牌可以包含对授权信息的引用，而不是直接编码信息。使用引用可能需要资源服务器与授权服务器之间的额外交互来将引用解析为授权信息。此类交互的机制不由本规范定义，但令牌 Introspection [RFC7662] 中定义了这样一种机制。

本文件未指定访问令牌的编码或内容；因此，关于保证访问令牌完整性保护手段的详细建议不在本规范范围内。访问令牌的编码和签名机制的一个示例在 JSON Web Token Profile for Access Tokens [RFC9068] 中描述。

为了处理访问令牌重定向，授权服务器在令牌中包含预期接收者（受众）的身份（通常是单个资源服务器或资源服务器列表）是很重要的。也将令牌的使用限制为特定作用域也是推荐的。

第 1.5 节提供了防止访问令牌泄露和保护客户端、资源服务器与授权服务器之间通信的机密性和完整性的信息。

### 7.1.3. 建议总结

### 7.1.3.1. 保护持有者令牌

客户端实现必须确保持有者令牌不会泄露给非预期方，因为它们将能够使用它们来访问受保护资源。这是使用持有者令牌时的主要安全考虑，也是以下所有更具体建议的基础。

### 7.1.3.2. 验证 TLS 证书链

客户端在向受保护资源发出请求时必须验证 TLS 证书链。如果不这样做，可能会启用 DNS 劫持攻击来窃取令牌并获得意外访问权限。

### 7.1.3.3. 始终使用 TLS (https)

客户端在使用持有者令牌发出请求时必须始终使用 TLS (https) 或等效传输安全。如果不这样做，会使令牌暴露于大量攻击，可能给攻击者带来意外访问权限。

### 7.1.3.4. 不要将持有者令牌存储在 HTTP Cookie 中

实现不得将持有者令牌存储在可以明文发送的 Cookie 中（这是 Cookie 的默认传输模式）。将持有者令牌存储在 Cookie 中的实现必须采取预防措施防止跨站请求伪造。

### 7.1.3.5. 颁发短期持有者令牌

授权服务器应颁发短期持有者令牌，特别是在向在 Web 浏览器或其他可能发生信息泄露的环境中运行的客户端颁发令牌时。使用短期持有者令牌可以减少它们被泄露的影响。

### 7.1.3.6. 颁发有作用域的持有者令牌

授权服务器应颁发包含受众限制的持有者令牌，将其使用限制于预期的资源服务器或资源服务器集。

### 7.1.3.7. 不要在页面 URL 中传递持有者令牌

持有者令牌不得通过页面 URL（例如，作为查询字符串参数）传递。相反，持有者令牌应通过 HTTP 消息头或消息体传递，并采取保密措施。浏览器、Web 服务器和其他软件可能无法充分保护浏览器历史记录、Web 服务器日志和其他数据结构中的 URL。如果持有者令牌通过页面 URL 传递，攻击者可能能够从历史数据、日志或其他不安全位置窃取它们。

### 7.1.4. 访问令牌权限限制

访问令牌关联的权限应限制为特定应用程序或用例所需的最少权限。这可以防止客户端超过资源所有者授权的权限。它还可以防止用户超过其各自安全策略授权的权限。权限限制也有助于减少访问令牌泄露的影响。

特别是，访问令牌应限制为某些资源服务器（受众限制），最好限制为单个资源服务器。为了实施这一点，授权服务器将访问令牌与某些资源服务器关联，每个资源服务器有义务验证，对于每个请求，随该请求发送的访问令牌是否用于该特定资源服务器。如果没有，资源服务器必须拒绝服务相应的请求。客户端和授权服务器可以使用本文件和 [RFC8707] 中分别指定的 scope 或 resource 参数来确定它们想要访问的资源服务器。

此外，访问令牌应限制为资源服务器上的某些资源和操作。为了实施这一点，授权服务器将访问令牌与相应的资源和操作关联，每个资源服务器有义务验证，对于每个请求，随该请求发送的访问令牌是否用于该特定资源上的特定操作。如果没有，资源服务器必须拒绝服务相应的请求。客户端和授权服务器可以使用 [RFC9396] 中指定的 scope 和 authorization_details 参数来确定这些资源和/或操作。

## 7.2. 客户端认证

根据客户端注册和凭证生命周期管理的整体过程，这可能影响授权服务器对特定客户端的置信度。

例如，动态注册客户端的认证不能证明客户端的身份，它只确保来自同一客户端实例的重复请求到达授权服务器。此类客户端可能在允许请求的作用域方面受到限制，或者可能有其他限制，例如较短的令牌生命周期。

相比之下，如果有一个已注册应用程序，其开发者身份已被验证，签署了合同，并被颁发仅在安全后端服务中使用的客户端密钥，授权服务器可能允许此客户端请求更敏感的作用域或颁发更持久的令牌。

## 7.3. 客户端冒充

如果机密客户端的凭证被盗，恶意客户端可以冒充该客户端并获取对受保护资源的访问权限。

授权服务器应强制执行明确的资源所有者认证，并向资源所有者提供有关客户端和请求的授权作用域及生命周期的信息。由资源所有者审查上下文中的信息并授权或拒绝请求。

授权服务器不应在不进行积极的资源所有者交互的情况下自动处理重复授权请求（除非已认证客户端或依赖其他措施以确保重复请求来自原始客户端而不是冒充者）。

### 7.3.1. 原生应用程序的冒充

如上所述，授权服务器不应在未征得用户同意或交互的情况下自动处理授权请求，但这是在可以保证客户端身份的情况下。这包括用户之前已批准给定客户端 ID 的授权请求的情况——除非可以证明客户端身份，否则应将请求视为未批准过之前的请求。

诸如声称的 https 方案重定向等措施可被授权服务器接受作为身份证明。某些操作系统可能提供适当的替代平台特定身份功能，可以接受。

### 7.3.2. 访问令牌权限限制

客户端应请求具有必要最小作用域的访问令牌。授权服务器在选择如何处理请求的作用域时应考虑客户端身份，并且可以颁发比请求更少作用域的访问令牌。

访问令牌关联的权限应限制为特定应用程序或用例所需的最少权限。这可以防止客户端超过资源所有者授权的权限。它还可以防止用户超过其各自安全策略授权的权限。权限限制也有助于减少访问令牌泄露的影响。

特别是，访问令牌应限制为某些资源服务器（受众限制），最好限制为单个资源服务器。为了实施这一点，授权服务器将访问令牌与某些资源服务器关联，每个资源服务器有义务验证，对于每个请求，随该请求发送的访问令牌是否用于该特定资源服务器。如果没有，资源服务器必须拒绝服务相应的请求。客户端和授权服务器可以使用 [RFC8707] 中分别指定的 scope 或 resource 参数来确定它们想要访问的资源服务器。

## 7.4. 客户端冒充资源所有者

资源服务器可能根据为该访问令牌颁发的资源所有者身份，或根据客户端凭证许可中的客户端身份做出访问控制决策。如果两种选项都可行，取决于实现细节，客户端的身份可能与资源所有者的身份混淆。例如，如果客户端能够在其注册期间选择自己的 client_id，恶意客户端可以将其设置为标识最终用户的值（例如，如果使用 OpenID Connect，则是 sub 值）。如果资源服务器无法正确区分颁发给客户端的访问令牌和颁发给最终用户的访问令牌，则客户端可能能够访问最终用户的资源。

如果授权服务器对客户端 ID 和用户标识符使用通用命名空间，导致资源服务器无法区分资源所有者授权的访问令牌和客户端自己授权的访问令牌，授权服务器不应允许客户端影响其 client_id 或任何其他可能导致与真正资源所有者混淆的声明。在无法避免的情况下，授权服务器必须为资源服务器提供其他手段来区分两种类型的访问令牌。

## 7.5. 授权码安全考虑

### 7.5.1. 授权码注入

授权码注入是一种攻击，客户端在其重定向 URI 中收到攻击者的授权码，而不是来自合法授权服务器的授权码。如果没有保护措施，客户端无法知道发生了攻击。授权码注入可能导致攻击者获取对受害者账户的访问权限，以及受害者意外获得对攻击者账户的访问权限。

### 7.5.1.1. 对策

为了防止向客户端注入授权码，客户端必须使用 code_challenge 和 code_verifier，并且授权服务器必须强制执行其使用，除非满足以下两个条件：

* 客户端是机密客户端。

* 在特定部署和特定请求中，授权服务器有合理保证客户端正确实现了 OpenID Connect nonce 机制。

在这种情况下，仍然建议使用和强制执行 code_challenge 和 code_verifier。

code_challenge 或 OpenID Connect nonce 值必须是特定于事务的，并安全地绑定到启动事务的客户端和用户代理。如果事务导致错误，必须选择新的 code_challenge 或 nonce 值。

依赖客户端验证 OpenID Connect nonce 参数意味着授权服务器无法确认客户端实际上已保护自己免受授权码注入攻击。如果攻击者能够将授权码注入客户端，客户端仍会交换注入的授权码并获取令牌，只会在验证 nonce 并发现不匹配后拒绝 ID 令牌。相比之下，授权服务器强制执行 code_challenge 和 code_verifier 参数可提供更高的安全性，因为授权服务器能够预先识别授权码注入攻击并避免首先颁发任何令牌。

历史说明：虽然 PKCE [RFC7636]（code_challenge 和 code_verifier 参数的创建地）最初设计为保护原生应用免受授权码泄露攻击，但所有类型的 OAuth 客户端，包括 Web 应用程序和其他机密客户端，都容易受到授权码注入攻击，而 code_challenge 和 code_verifier 机制可以解决这些问题。

### 7.5.2. 授权码重用

如果授权码能够使用多次，则可能发生多种类型的攻击。

如第 4.1.3 节所述，当收到带有已用于颁发访问令牌的授权码的第二个有效请求时，授权服务器必须拒绝令牌请求并撤销任何已颁发的令牌。如果攻击者能够在合法客户端之前泄露并使用授权码，攻击者将获取访问令牌，而合法客户端则不会。撤销任何已颁发的令牌意味着攻击者的令牌将被撤销，阻止攻击继续进行。

但是，授权服务器仅应在包含授权码的请求也有效时才撤销已颁发的令牌，包括任何其他参数（如 code_verifier 和客户端认证）。当收到包含无效参数的重放授权码时，授权服务器不应撤销任何已颁发的令牌。如果这样做，会为能够获取授权码但在发送无效授权码请求（在合法客户端之前）时无法获取客户端认证或 code_verifier 的攻击者创造拒绝服务机会，从而在其发出有效请求后撤销合法客户端的令牌。

### 7.5.3. HTTP 307 重定向

如果重定向可能包含用户凭证的请求，授权服务器不得使用 307 状态码（[RFC9110] 第 15.4.8 节）进行重定向。如果对此类请求使用 HTTP 重定向（而不是例如 JavaScript），AS 应使用状态码 303（"See Other"）。

在授权端点，典型协议流程是 AS 提示用户在表单中输入凭证，然后（使用 POST 方法）提交回授权服务器。AS 检查凭证，如果成功，则将用户代理重定向到客户端的重定向 URI。

如果使用状态码 307 进行重定向，用户代理会将用户凭证通过 POST 请求发送到客户端。

这会将敏感凭证泄露给客户端。如果客户端是恶意的，它可以使用这些凭证在 AS 冒充用户。

该行为可能对开发者来说出乎意料，但在 [RFC9110] 第 15.4.8 节中定义。此状态码不要求用户代理将 POST 请求重写为 GET 请求，从而丢弃 POST 请求内容中的表单数据。

在 HTTP [RFC9110] 中，只有状态码 303 明确强制将 HTTP POST 请求重写为 HTTP GET 请求。对于所有其他状态码（包括流行的 302），用户代理可以选择不将 POST 重写为 GET 请求，从而将用户凭证泄露给客户端。（然而，实际上大多数用户代理只会对 307 重定向显示此行为。）

## 7.6. 确保端点真实性

中间人攻击的风险通过强制使用通道安全机制（如 [RFC8446]）与授权和令牌端点通信来缓解。详见第 1.5 节。

## 7.7. 凭证猜测攻击

授权服务器必须防止攻击者猜测访问令牌、授权码、刷新令牌、资源所有者密码和客户端凭证。

攻击者猜测生成令牌（以及其他不面向最终用户处理的凭证）的概率必须小于或等于 2^(-128)，并应小于或等于 2^(-160)。

授权服务器必须使用其他方式来保护面向最终用户使用的凭证。

## 7.8. 钓鱼攻击

广泛部署此和类似协议可能导致最终用户习惯于被重定向到要求输入密码的网站。如果最终用户在输入凭证之前不注意验证这些网站的真实性，攻击者将能够利用此做法窃取资源所有者的密码和其他可钓鱼凭证（如 OTP）。

服务提供商应尝试教育最终用户注意钓鱼攻击带来的风险，并应提供使最终用户能够轻松确认其站点真实性的机制，例如使用抗钓鱼认证器，因为抗钓鱼认证器仅在平台成功验证站点来源后才向用户提供登录到某个站点的凭证。客户端开发者应考虑其与用户代理交互（例如外部、嵌入式）的安全影响，以及最终用户验证授权服务器真实性的能力。

有关缓解钓鱼攻击风险的更多详细信息，请参见第 1.5 节。

## 7.9. 跨站请求伪造

攻击者可能尝试在受害者设备上向合法客户端的重定向 URI 注入请求，例如导致客户端访问攻击者控制下的资源。这是称为跨站请求伪造（CSRF）的攻击变体。

传统对策是客户端传递一个随机值（也称为 CSRF Token），在 state 参数中链接请求与重定向 URI 到用户代理会话，如所述。此对策在 [RFC6819] 第 5.3.5 节中有详细描述。code_verifier 参数或 OpenID Connect nonce 值提供相同的保护。

当使用 code_verifier 而不是 state 或 nonce 进行 CSRF 保护时，重要的是要注意：

* 客户端必须确保 AS 支持其打算使用的 code_challenge_method。如果授权服务器不支持请求的方法，则必须使用 state 或 nonce 进行 CSRF 保护。

* 如果 state 用于携带应用程序状态，且其内容完整性令人担忧，客户端必须保护 state 免受篡改和交换。这可以通过将 state 内容绑定到浏览器会话和/或签名/加密状态值 [I-D.bradley-oauth-jwt-encoded-state] 来实现。

因此，AS 必须提供一种方式来检测其支持的代码挑战方法，要么通过根据 [RFC8414] 的 AS 元数据，要么提供特定于部署的方式以确保或确定支持。

## 7.10. 点击劫持

如 [RFC6819] 第 4.4.1.9 节所述，授权请求容易受到点击劫持攻击，也称为 UI 重定向。在这种情况下，攻击者将授权端点用户界面嵌入到无害上下文中。用户相信与该上下文交互，例如单击按钮，却无意中与授权端点用户界面交互。反向也可以实现：用户相信与授权端点交互，却无意中在攻击者提供的覆盖在原始用户界面上的输入字段中输入密码。点击劫持攻击可以设计得让用户几乎注意不到攻击，例如使用几乎不可见的 iframe 覆盖在其他元素之上。

攻击者可以使用此向量获取用户的认证凭证，更改授予客户端的访问作用域，并可能访问用户的资源。

授权服务器必须防止点击劫持攻击。[RFC6819] 中描述了多种对策，包括使用 X-Frame-Options HTTP 响应头字段和框架破坏 JavaScript。除此之外，授权服务器还应使用内容安全策略（CSP）级别 2 [CSP-2] 或更高版本。

为了有效，CSP 必须用于授权端点，以及（如适用）用于认证用户和授权客户端的其他端点（例如设备授权端点、登录页面、错误页面等）。这可以防止在使用支持 CSP 的用户代理中进行未授权来源的框架设置。客户端可能允许被某些不同于其重定向端点使用的来源框架。因此，授权服务器应允许管理员为特定客户端配置允许的来源和/或允许客户端动态注册这些。

使用 CSP 允许授权服务器在单个响应头字段中指定多个来源，并使用灵活模式进行约束（详见 [CSP-2]）。该标准的级别 2 通过使用限制框架来源的政策（使用 frame-ancestors）以及限制允许在 HTML 页面上执行的脚本来源的政策（使用 script-src）提供了防止点击劫持的强大机制。以下列表中显示了该策略的规范性示例：

```
HTTP/1.1 200 OK
Content-Security-Policy: frame-ancestors https://ext.example.org:8000
Content-Security-Policy: script-src 'self'
X-Frame-Options: ALLOW-FROM https://ext.example.org:8000
...
```

由于某些用户代理不支持 [CSP-2]，除非此类传统用户代理明确不受授权服务器支持，否则此技术应与其他技术结合使用，包括 [RFC6819] 中描述的那些。即使在这种情况下，仍应采用其他对策。

## 7.11. 注入和输入验证

当应用程序未经清理就使用输入或其他外部变量时，会发生注入攻击。这可能允许攻击者访问应用程序设备或其数据、导致拒绝服务或引入广泛的恶意副作用。

授权服务器和客户端必须将收到的参数视为潜在的恶意外部输入并应用适当的保护，特别是 state 和 redirect_uri 参数的值。

## 7.12. 开放重定向

开放重定向器是一个端点，可将用户的浏览器转发到从查询参数获取的任意 URI。此类端点有时会被实现，例如在用户被重定向到外部网站之前显示消息，或将用户重定向回他们中断前打算访问的 URL（例如，被登录提示中断）。

当 AS 或客户端具有开放重定向器时，可能发生以下攻击。

### 7.12.1. 客户端作为开放重定向器

客户端不得暴露开放重定向器。攻击者可能使用开放重定向器生成指向客户端的 URL，并利用它们来泄露授权码，如 [RFC9700] 第 4.1.1 节所述。另一个滥用案例是生成看起来指向客户端的 URL。这可能诱骗用户信任 URL 并在浏览器中跟随它。这可用于钓鱼。

为了防止开放重定向，客户端只应在目标 URL 在白名单中或可以验证请求的来源和完整性时重定向。开放重定向对策由 OWASP [owasp_redir] 描述。

### 7.12.2. 授权服务器作为开放重定向器

与客户端一样，攻击者可能利用用户对授权服务器（特别是其 URL）的信任来执行钓鱼攻击。OAuth 授权服务器定期将用户重定向到其他网站（客户端），但必须安全地执行此操作。

第 4.1.2.1 节已经通过声明授权服务器在 client_id 和 redirect_uri 组合无效的情况下不得自动重定向用户代理来防止开放重定向。

但是，攻击者也可以利用正确注册的重定向 URI 执行钓鱼攻击。例如，攻击者可以通过动态客户端注册 [RFC7591] 注册客户端并执行以下攻击之一：

1. 故意发送错误的授权请求，例如使用无效的作用域值，从而指示授权服务器将用户代理重定向到其钓鱼站点。

2. 故意使用攻击者控制的 client_id 和 redirect_uri 发送有效授权请求。用户认证后，授权服务器提示用户授予请求同意。如果用户注意到请求有问题并拒绝请求，授权服务器仍然会将用户代理重定向到钓鱼站点。在这种情况下，无论用户采取什么操作，用户代理都会被重定向到钓鱼站点。

3. 故意使用攻击者控制的 client_id 和 redirect_uri 发送有效静默认证请求（prompt=none）。在这种情况下，授权服务器会自动将用户代理重定向到钓鱼站点。

授权服务器必须采取预防措施来防止这些威胁。授权服务器必须始终首先认证用户，并且在重定向用户之前（静默认证用例除外）在需要时提示用户输入凭证。根据其风险评估，授权服务器需要决定是否可以信任重定向 URI。它可以考虑内部或通过某些外部服务进行的 URI 分析来评估 URI 背后的内容可信度和可信赖性，以及重定向 URI 和其他客户端数据的来源。

如果信任重定向 URI，授权服务器只应自动重定向用户代理。如果 URI 不受信任，授权服务器可以通知用户并依靠用户做出正确决定。

## 7.13. 传输安全

在某些部署中，包括使用负载均衡器的部署，到资源服务器的 TLS 连接在实际提供资源的服务器之前的终端。这可能使令牌在 TLS 连接终端的前端服务器和提供资源的后端服务器之间不受保护。在此类部署中，必须采取充分措施确保前端和后端服务器之间访问令牌的机密性；加密令牌就是这样一种可能的措施。

有关更多信息，请参见 [RFC9110] 第 17.2 节。

## 7.14. 授权服务器混消缓解

混消是一种针对 OAuth 客户端与两个或多个授权服务器交互且至少有一个授权服务器在攻击者控制下的场景的攻击。例如，如果攻击者使用动态注册在自己的授权服务器注册客户端，或者如果授权服务器被泄露，则可能发生这种情况。

当 OAuth 客户端只能与一个授权服务器交互时，不需要混消防御。然而，在 OAuth 客户端与两个或多个授权服务器交互的场景中，客户端必须防止混消攻击。以下讨论两种不同的方法。

对于两种防御，客户端必须为每个授权请求存储它们发送授权请求到的颁发者，将此信息绑定到用户代理，并检查授权响应是否来自正确的颁发者。客户端必须确保后续的访问令牌请求（如果适用）被发送到相同的颁发者。颁发者通过关联的元数据，作为授权端点和令牌端点组合的抽象标识符，用于流程中。如果颁发者标识符不可用，例如，如果既未使用 OAuth 2.0 授权服务器元数据 [RFC8414] 也未使用 OpenID Connect Discovery [OpenID.Discovery]，则可以使用不同的唯一标识符或元组本身来代替。为了简洁起见，在下文中，此类特定于部署的标识符将被归入颁发者（或颁发者标识符）。

注意：仅存储授权服务器 URL 不足以识别混消攻击。攻击者可能将其未受损的 AS 授权端点 URL 声明为"自己的"AS URL，但声明自己控制下的令牌端点。

有关多种混消攻击类型的详细说明，请参见 [RFC9700] 第 4.4 节。

### 7.14.1. 通过颁发者标识进行混消防御

此防御要求授权服务器在授权响应中向客户端发送其颁发者标识符。收到授权响应时，客户端必须将收到的颁发者标识符与存储的颁发者标识符进行比较。如果不匹配，客户端必须中止交互。

此颁发者标识符可以以不同方式传输到客户端：

* 例如，颁发者信息可以通过可选响应参数 iss 传输（见第 4.1.2 节）。

* 当使用 OpenID Connect 并在授权响应中返回 ID 令牌时，客户端可以评估 ID 令牌中的 iss 声明。

在这两种情况下，必须按照 [RFC9207] 评估 iss 值。

虽然此防御可能需要使用附加参数来传输颁发者信息，但它是一种针对混消的强大且相对简单的防御。

### 7.14.2. 通过不同重定向 URI 进行混消防御

对于此防御，客户端必须为其交互的每个颁发者使用不同的重定向 URI。

客户端必须通过将颁发者的不同重定向 URI 与接收授权响应的 URI 进行比较来检查授权响应是否来自正确的颁发者。如果不匹配，客户端必须中止流程。

虽然此防御建立在现有 OAuth 功能之上，但它不能用于客户端仅为许多不同颁发者注册一次（例如，在某些开放银行方案中）的场景，并且由于与客户端注册的紧密集成，更难自动部署。

此外，攻击者可能通过使用客户端分配给攻击者 AS 的重定向 URI 在"诚实"AS 上注册新客户端来规避此防御提供的保护。攻击者可以像上述那样运行攻击，但用他新创建的客户端的客户端 ID 替换客户端 ID。

因此，此防御应仅在其他选项不可用时使用。
